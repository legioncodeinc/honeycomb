# Wiki-summaries conventions (PRD-017) — READ BEFORE FILLING THE WAVE-2 STUB

Wave 1 (017a summary-worker + the contracts/seams) established these. Wave 2
(017b synthesis) follows them **verbatim**. The point of the seam wiring is that the
Wave-2 Bee (`retrieval-worker-bee`) edits **only `synthesis.ts` + `synthesis.test.ts`**
and never touches `contracts.ts`, `worker.ts`, the catalog, or `server.ts`.

---

## 1. The thesis (every rule descends from this)

- **Collapse verbose session traces → an AI-written wiki summary written to the
  `memory` table → recall ranks DOCUMENTS, not thousands of raw `sessions` rows.** 017a
  writes one summary per session at `/summaries/<userName>/<sessionId>.md`; 017b
  synthesizes the `MEMORY.md` + thread heads that link them.
- **The hook SIGNALS the daemon; the daemon owns the ONLY DeepLake connection.** No
  hook and no thin-client code opens DeepLake. The worker runs INSIDE the daemon (port
  3850, the only DeepLake client) and reads `sessions` + writes `memory` through the
  daemon's own `StorageQuery`. The hook half (final + periodic triggers) is a thin
  client that signals the daemon. (See §2.)

## 2. Daemon-only storage (FR-1 / the thin-client invariant)

Every `sessions` read and `memory` write goes through the daemon-side seams
(`SessionEventFetcher` / `SummaryStore`) over the daemon's `StorageQuery`. The worker
NEVER re-opens DeepLake. `tests/daemon/storage/invariant.test.ts` keeps this honest:
summaries live under `src/daemon/`, and the hook trigger signals the daemon over 3850.
**Do not import `createStorageClient` / anything under `daemon/storage` except the pure
`sql.ts` helpers** in any thin-client (hook) code.

## 3. SELECT-before-INSERT keyed on `path` — NEVER an in-place UPDATE (a-AC-6 / D-6)

**a-AC-6 LITERALLY requires SELECT-before-INSERT keyed on `path`, not an in-place
UPDATE.** The per-session summary is written EXACTLY ONCE:

> probe `memory` for the `path` → insert iff absent → re-verify. An existing real
> summary at the path means it already landed (exactly-once). The prior in-progress
> placeholder is REMOVED (a guarded `DELETE … AND description = 'in progress'`) just
> before the insert — never mutated.

The reasons are the same ones `memory_jobs`, `ontology/supersede.ts`,
`sources/lifecycle.ts`, `skillify/skills-write.ts`, and `pipeline/graph-persist.ts` all
hit and solved live:

- An **in-place UPDATE** on this backend coalesces rapid writes and serves reads from
  segments of differing freshness that flap non-monotonically — a by-`path` `SET` can
  never converge.
- A **live read of a just-written row UNDER-reports** (a stale segment), so live
  read-backs are **poll-convergent** (poll + take the row that is present; a read can
  miss the write but never invents one).

`SummaryStore` (in `worker.ts`) has **NO `update` method** by construction. **Do not add
an in-place UPDATE anywhere in this subsystem.** The unit test asserts NO UPDATE was
emitted (like the sources / skillify tests).

## 4. SQL safety (FR-5)

Every dynamic fragment routes through the 002b helpers: `sqlIdent` (identifiers),
`sLiteral` / `eLiteral` via the `val.*` constructors (values). The embedding column is
either a pre-validated `serializeFloat4Array` numeric fragment or a raw `NULL` —
`val.raw`, never hand-quoted. Every write goes through the heal-aware
`selectBeforeInsert` / `withHeal` primitives (→ guarded `buildInsert`). `npm run
audit:sql` scans `src/daemon` and fails CI on a raw interpolation.

## 5. The 017a posture the Wave-2 Bee inherits (do not regress)

- **Per-session lock (a-AC-4 / D-2):** an atomic `O_EXCL` lock file keyed by
  `sessionId` (`createFileSessionLock`, adapted from skillify's `createFileWorkerLock`
  but PER-SESSION). A second concurrent trigger for the same session is SUPPRESSED.
  Released in `finally`.
- **Retry-on-empty + placeholder cleanup (a-AC-3 / D-3):** DeepLake read lag → if the
  event fetch finds NONE, retry with LINEAR backoff up to `WorkerConfig.retryLimit` at
  `backoffMs`. On the final give-up, REMOVE the in-progress placeholder (never strand
  it). The placeholder is itself a SELECT-before-INSERT row whose `description` is
  `'in progress'`; the removal is guarded by that marker so a concurrent real summary is
  never clobbered.
- **Gate-CLI, no API key (a-AC-1 / a-AC-2 / D-4):** the summary is generated by SHELLING
  OUT to the host harness's own CLI (`SummaryGenCli` seam), a `spawn` with an **args
  array, never a shell string** (`shell:false`, so a transcript can't command-inject),
  bounded by `gateTimeoutMs` with a SIGTERM kill. The subprocess env sets
  **`HONEYCOMB_WIKI_WORKER=1` + `HONEYCOMB_CAPTURE=false`** so the gate call does not
  trigger its OWN capture loop.
- **Embed is NON-FATAL (a-AC-5 / D-5):** `EmbedClient.embed()` → the 768-dim
  `summary_embedding`; a THROW (or disabled / wrong-dim) → store NULL, the write STILL
  succeeds. `embedNonFatal` is the wrapper.
- **Redaction at the boundary (PRD-017 Wave 3 SECURITY):** every event text is scrubbed
  with skillify's `redactSecrets` before it reaches the gate prompt, so a pasted
  credential never lands in a summary. Imported from `skillify/miner.ts`, not forked.

## 7. The seams Wave 2 inherits (pinned contracts)

- `SummaryTrigger` — `{ kind: "final", event } | { kind: "periodic", reason, count }`.
  The final (`Stop`/`SessionEnd`/`session_shutdown`) + periodic (messages/hours)
  triggers the hook signals the daemon with.
- `WorkerConfig` — `{ retryLimit, backoffMs, gateTimeoutMs }`. The retry-on-empty +
  gate-timeout tuning a run serializes. Defaults: `DEFAULT_WORKER_CONFIG`.
- `SummarySession` — `{ sessionId, userName, path, agentId? }`. The session identity
  (lock key + path segments + event-fetch grouping key + gate-CLI selection).
- `SessionEvent` / `SessionEventFetcher` — one fetched `sessions` event + the ONE-attempt
  fetch seam (the worker drives the retry loop on top).
- `SummaryGenCli` — `{ run(prompt): Promise<string> }`. The host-CLI shell-out → markdown
  (017a). Fake via `createFakeSummaryGenCli(markdown)`.
- `EmbedClient` — reused from PRD-005b (`embed(text) → number[] | null`); a throw is
  non-fatal.
- `SummaryLock` — `{ acquire(sessionId): SummaryLockHandle | null }`. The per-session
  `O_EXCL` lock (a-AC-4). Real impl: `createFileSessionLock(baseDir?)`.
- `SummaryStore` — `{ writePlaceholder, removePlaceholder, writeSummary }`. The daemon's
  `memory` SELECT-before-INSERT path (a-AC-1 / a-AC-3 / a-AC-6). **NO `update`** by
  construction. Real impl: `createSummaryStore(storage, scope, resolveTable?)`.

## 8. Live-test isolation (the proven technique)

The opt-in live itest (`tests/integration/summary-write-live.itest.ts`) SELECT-before-
INSERTs a summary at a `/summaries/…` path to the REAL backend, reads it back
**poll-convergent**, and asserts a re-run does NOT double (exactly-once). Isolation is
**native throwaway tables**: inject `resolveTable` into `createSummaryStore` so the heal
CREATEs a per-run `ci_summaries_<runid>` table directly — **NOT a SQL-string proxy**
(which races the heal's CREATE/introspect/ALTER and corrupts a fresh table).
`queryTimeoutMs: 120_000`; DROP the table in `afterAll`. Do NOT run it locally (no creds)
— the orchestrator runs it.

## 9. Deferred daemon-assembly + hook wiring

The summary **worker** (the `memory_jobs` job that runs `runSummaryWorker` on a trigger)
and the **hook signal** (the final + periodic triggers → daemon via `turn-counters`'
periodic threshold) are mounted by the daemon-assembly step (the same deferred-assembly
posture as `skillify` / `sources/api.ts` / `secrets/api.ts`). Wave 1 does NOT edit
`server.ts` or the worker registry — it ships the worker core + the seams. The assembly
step constructs the real `{ lock, fetcher, gate, embed, store }` deps at daemon
construction and routes a trigger to `runSummaryWorker`.
