Re-drive design + limitation: the terminal `memory_controlled_write` job payload carries the ALREADY-RESOLVED proposal + fact material + scope envelope (org/workspace/agent_id) — NOT the built `RowValues` row. Per the PRD's documented escape hatch, `redriveControlledWritePayload` rebuilds the resolved write by RE-RUNNING the controlled-write proposal path (`applyControlledWrite`, the SAME single-sourced path the live stage runs), NOT by hand-reconstructing the row. On a healthy backend it commits directly (idempotent — a memory a prior attempt landed is `deduped`, no duplicate); during a still-degraded window it defers into the durable `memory_outbox` (`deferred`) for the drainer to land on recovery — either way the dropped memory is recovered. Limitation (documented): the re-drive deps OMIT `onOutcome` (graph-persist fan-out) and `onConflict` (post-commit conflict detection) — a re-driven memory lands its durable `memories` row (the BUG-04b goal) but its entity-graph edges + conflict projection reconcile via the idempotent pollinating pass rather than inline. Re-driven ADDs also carry a NULL embedding only if embeddings are unavailable (the real `embed.client` is wired in production), exactly the 080a/capture degrade.

080c design notes (the c-AC-2 dedup-preservation nuance): the memory commit is DEDUP-GATED, so — unlike the capture coalescer (a bare `appendOnlyInsertMany`, no dedup) — a naive multi-row append would BYPASS the `content_hash` dedup and risk DUPLICATE memories. The chosen design is the PRD's option (a) for ADD groups + option (b)'s per-row safety everywhere else: an ADD group of ≥2 members whose hashes are ALL present AND DISTINCT drains through ONE batched `content_hash IN (…)` probe (`commitControlledWriteMany`) that filters already-landed hashes to `deduped` (no insert — idempotent), then multi-row appends ONLY the not-present rows; a group with an IN-GROUP duplicate hash (two pending ADDs of the same fact) is NOT provably safe to batch (the DB-side probe cannot dedup two rows against each other) so it falls back to the per-row `commitControlledWrite` where the sequential dedup-probe-then-append catches the sibling; UPDATE/DELETE (a NON-idempotent per-key version-bump) is NEVER coalesced (kept per-row) — `groupKey` includes the action so an ADD never fuses with a version-bump despite the shared column signature. A failed group backs off / dead-letters EACH member independently (per-row accounting via the shared `settleFailure`), so no row is lost or double-committed. Per-pass cap reconciliation (c-AC-3): the 080a per-pass lease cap (`maxDrainPerPass`, the `leaseDue LIMIT`) and the new back-pressure knob are the SAME concept, so they were unified into ONE authoritative `maxDrainPerInterval` (mirroring how 079c unified `drainBatch → maxDrainPerInterval`); the old constant name is retained only as a `@deprecated` alias.

W-1 close-out (memory-formation counting): a drain-recovered commit now feeds the SAME `MemoryFormationTracker` the live stage feeds, so `memoryFormation.committedSinceBoot` climbs THROUGH a degraded window (BUG-04's Verify criteria).
