## Critical Issues (must fix)

### C-1. Conflict detection is never invoked on any live path (`memory_conflicts` is never populated in production)

**Evidence:**
- `detectAndProject` (the detect → resolve → project-into-`memory_conflicts` entry point) is defined at `src/daemon/runtime/memories/conflict-resolve.ts:528`. Its ONLY reference outside its own file is a barrel re-export at `src/daemon/runtime/memories/index.ts:135`. There is **no call site** in any production module.
- `src/daemon/runtime/pipeline/controlled-writes.ts` (the decision-stage handler) contains **zero** conflict references (`grep -E "conflict|Conflict"` returns empty).
- `src/daemon/runtime/memories/conflicts-api.ts` mounts **resolve-only** (`POST /api/memories/conflicts/:id/resolve`); `lifecycle-api.ts` is **reads-only**. Neither triggers detection. There is no detection-trigger route (contrast the stale-ref path, which IS wired - see below).

**Why this is Critical - the PRD explicitly mandates the wiring that is missing:**
- `prd-058b-...-conflict-resolution.md:160` - "Detection runs in the **decision stage** (over the candidate set already fetched per extracted fact) **and as a maintenance-worker pass**."
- `prd-058b-...:205` (Files Touched) - "the **decision-stage worker** (per memory-pipeline.md, the controlled-writes / decision handler) - **call the detector on the existing candidate set**."

Because nothing populates `memory_conflicts`, the recall-time κ suppression source (`createConflictSuppressionSource`, correctly wired at `src/daemon/runtime/assemble.ts:876,882`) reads the open-conflict projection and finds nothing. On a live daemon, two memories that assert contradictory outcomes **both surface in a single recall result set** - the exact behavior parent AC **IDX-2** forbids ("two contradictory never both appear; loser suppressed; recorded to history"). The unit tests pass only because they invoke `detectAndProject` directly with injected candidates; they structurally cannot catch this assembly gap (project memory: "Dogfood surfaces integration bugs"; "Completed ≠ live (deferred assembly)").

**Scope note:** this is NOT a credential-blocked item. The detector wiring into `controlled-writes.ts` (or a maintenance route mirroring `stale-ref-api.ts`) is pure code assembly, independent of `HONEYCOMB_DEEPLAKE_TOKEN`. The stale-ref sibling demonstrates the wiring was achievable: `runStaleRefDiagnostic` IS reachable via `src/daemon/runtime/maintenance/stale-ref-api.ts:152`. Conflict detection simply was not given an equivalent live entry point.

**Recommended remediation (for the orchestrator to route - do NOT fix here):**
Wire `detectAndProject` into the decision-stage handler in `src/daemon/runtime/pipeline/controlled-writes.ts` over the candidate set the decision stage already fetches (the PRD's named integration point), and/or add a maintenance-trigger route mirroring `stale-ref-api.ts`. Then add a live-path test that stores a fact, stores its negation through the real write pipeline, and asserts a row lands in `memory_conflicts` and the loser is suppressed at recall.

## Design-Reconciliation Verdict

**The flagged scope deviation is ACCEPTABLE convention-alignment, NOT a tenancy gap. Confirmed sound.**

The ledger flagged (58e.note): PRD-058e's `memory_access` sketch listed explicit `org`/`workspace`/`agent_id` columns, but the implementer used the codebase engine-table convention (org/workspace at the storage-partition layer, `agent_id` carried). My independent verification:

- The codebase has a **documented three-way scope taxonomy** in `src/daemon/storage/catalog/types.ts:63-74`: `agent` (engine tables carry `agent_id` + `visibility`; org/workspace isolation comes from the storage partition / QueryScope), `tenant` (cross-cutting tables carry explicit `org_id` + `workspace_id`), `none` (transitively scoped audit/history).
- The pre-existing `memories` engine table uses `scope: "agent"` (`memories.ts:163`) and carries only `agent_id` + `visibility` (`memories.ts:76-77`) - NO org/workspace columns. This is the established convention for every engine table (`memories`, `sessions`, `memory`, `skills`).
- The new tables follow it exactly: `memory_access`, `memory_calibration` (`memory-lifecycle.ts:144-152`, `scope: "agent"`) and `memory_conflicts` (`memory-conflicts.ts:188`, `scope: "agent"`), each carrying `agent_id` + `visibility`.

**Why this is correct, not a gap:** Had the implementer added literal `org`/`workspace` columns, these three tables would have been the ONLY engine tables in the catalog inconsistent with the partition model - a real architecture defect. The `tenant` scope exists precisely for tables that need explicit org/workspace columns; the implementer correctly chose NOT to use it because cross-workspace conflict/access detection is **explicitly out of scope** (parent index line 52; 058b "Out of scope"). The storage-partition boundary IS the conflict/access boundary, so the partition supplies org/workspace and the row carries `agent_id`. A row is reachable transitively by its `memory_id`. **Tenancy isolation is fully preserved.** `memory_conflicts` scoping is sound by the same reasoning, with the added safety that its open-conflict projection pins MAX(version) per pair so a reversed/superseded earlier version never leaks (`memory-conflicts.ts:203-223`).
