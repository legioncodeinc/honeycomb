## Suggestions (consider improving)

- [ ] **Pan delta does not compensate for the rendered-element aspect ratio**, `src/dashboard/web/pages/graph.tsx:321-329`

  `onPointerMove` converts a client-pixel delta to base-box units using only `transform.scale` (`dx = (clientX - x) / scale`). The base box is 1600×1000 (1.6:1) but the canvas container is `flex:1 … height:70vh`, so on most viewports the rendered SVG is not 1.6:1 and a drag tracks the cursor slightly faster on one axis. The page comment already flags this as "approximate — good enough for pan," and AC-2 asks only that "dragging the background pans" (satisfied). Consider scaling `dx`/`dy` by the measured client width/height vs the viewBox extent for 1:1 cursor tracking if pan precision is later reported as off.

- [ ] **Search recomputes the full layout on every keystroke**, `src/dashboard/web/pages/graph.tsx:543-555`

  `onSearch` calls `layout(visible.nodes, visible.edges, GRAPH_VIEW)` on each change to find the matched node's position for centering. The canvas already memoizes `positions` via `useMemo` keyed on `graph` (`graph.tsx:311`); `onSearch` could read from a shared memoized position map instead of recomputing. `layout` is O(n) and deterministic, so this is harmless at expected snapshot sizes (matches OQ-3's "don't pre-optimize" guidance) — a readability/efficiency nicety, not a correctness issue.
