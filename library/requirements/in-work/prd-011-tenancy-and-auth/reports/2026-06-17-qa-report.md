## Suggestions (consider improving)

### S-1 — 011c PRD prose still names the stale `operator` role

**Severity:** Suggestion

**File:** `library/requirements/in-work/prd-011-tenancy-and-auth/prd-011c-tenancy-and-auth-modes-rbac.md`

**Location:** Lines 8 (`Goals`), 27 (user story), 36 (FR-4), and the AC table prose reference.

**Evidence:** The PRD-011c document reads:
- FR-4: `"admin (everything…), operator (remember, recall…), agent (…), readonly (recall only)"`
- User story: `"As an operator, I want a hybrid mode…"`

The frozen Wave-1 contract at `src/daemon/runtime/auth/contracts.ts:82` is:
```ts
export const ROLES = Object.freeze(["admin", "member", "readonly", "agent"] as const);
```

The ledger D-1 is authoritative (`member` is the binding), and `rbac.ts:24-27` explicitly documents the reconciliation:
```
// NOTE on the role vocabulary: the FROZEN Wave-1 contract ... pins the four roles to
// `admin | member | readonly | agent`. The 011c PRD *prose* still names a stale
// `operator` role; the index/ledger reconciled that to `member`, and the frozen
// `Role` type is authoritative.
```

The plan itself (`prd-011c`) is the document that needs updating so future readers do not face the discrepancy. The implementation is correct; the stale prose is the only gap.

**Recommended remediation:** In `prd-011c-tenancy-and-auth-modes-rbac.md`, replace all occurrences of `operator` with `member` in FR-4, the user story, and the AC table text. This is a `library-worker-bee` doc-only change with no code impact.
