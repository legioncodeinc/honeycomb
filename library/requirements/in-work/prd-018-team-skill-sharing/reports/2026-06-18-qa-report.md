## Suggestions (consider improving)

- **L1 follow-up: add `isSafeDirSegment` check to `normalizeEntry`**, `src/daemon-client/skillify/manifest.ts:86`

  The security report documents this as a defense-in-depth hardening recommendation. `normalizeEntry` currently accepts any non-empty `dirName` string at parse time; `isSafeDirSegment` is only called at the two destructive use-sites. Adding the check to `normalizeEntry` would fail closed at the parse boundary, so a poisoned record never survives `manifest.read()` at all. With H1/H2 use-site floors in place this is not exploitable, making it a Suggestion rather than a Warning. Recommended for a follow-up PR, not a blocker.

  ```typescript
  // In normalizeEntry (manifest.ts:86): add after the empty-string guard
  if (!isSafeDirSegment(dirName)) return null; // parse-boundary defense-in-depth
  ```

- **a-AC-6 test assertion could be tighter**, `tests/daemon/runtime/skillify/publish-endpoint.test.ts:114-121`

  The a-AC-6 test asserts (1) `fake.requests.length > 0` and (2) all requests are SELECT statements. This is a genuine, non-tautological assertion — it proves the statement traverses the `FakeDeepLakeTransport` (i.e., the daemon storage path). However, it does not assert that the `scope` partition (`{ org: "o1", workspace: "ws1" }`) is present on the dispatched request. The `FakeDeepLakeTransport` records the full `TransportRequest` including scope; a future iteration could add `expect(fake.requests[0]?.scope).toEqual(SCOPE)` to make the scope-carrying aspect explicit rather than implied. Acceptable as-is because the other endpoint tests (`a-AC-1`, `a-AC-5`) exercise the same code path and the invariant test (3/3 green) confirms no bypass.

- **`src/cli/index.ts` bin dispatch deferred**, referenced by CONVENTIONS.md §13

  The `skill scope` / `skill unpull` commands are fully implemented and tested but not yet wired into the bundled `honeycomb` bin (`src/cli/index.ts` remains the version-print stub). This is intentional and honestly documented (CONVENTIONS.md §13, ledger watchdog). A follow-up PR should mount `skill.ts`, `skillify.ts`, and `org.ts` on the bin together so the deferred posture resolves cleanly in one wiring wave. Not a blocker.
