## Suggestions (consider improving)

- [ ] **Baseline `//ndcg` note carries two different live nDCG numbers; a one-line reconciliation would help future readers**, `eval/recall-baseline.json:4` and `library/requirements/in-work/prd-047-retrieval-quality-upgrades/reports/2026-06-24-reranker-activation-eval.md:17`

  The baseline note records the dedup-on dedup-invariant live nDCG@10 = `0.609`, while the reranker-eval report's table lists the f-AC-3 baseline run at `0.596` (pre-dedup, clone-summing) and an embedding-cosine run at `0.584`. Both are correct and the baseline note already explains the 0.596→0.609 instrument-artifact correction, but the two documents read in isolation could confuse a future tuner. Consider a single cross-reference line in the reranker report pointing at the baseline note's dedup-invariance explanation. Non-blocking; the committed floor (`0.55` → enforced `0.50`) sits conservatively below every measured value.

- [ ] **The `llm` reranker strategy is enum-valid but unimplemented; it silently falls through to RRF order**, `src/daemon/runtime/memories/recall.ts:805`

  `rerankHits` returns the RRF order for any `strategy !== "embedding-cosine"`, so `reranker.strategy: "llm"` is a no-op rather than an error. This is the documented plan intent ("`llm` … a measured follow-up", 047b scope) and is honestly commented in code, so it is correct-as-specified. A future Suggestion only: when the `llm` branch lands, ensure an explicitly-requested-but-unbuilt strategy is observable (a debug log) rather than silently behaving as `none`.
