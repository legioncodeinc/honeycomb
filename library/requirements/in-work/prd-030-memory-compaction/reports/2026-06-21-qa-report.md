## Critical Issues (must fix)

### C-1 — `epistemic_assertions` is mapped to the WRONG key column → compaction silently no-ops on it (reopens AC-6)

**Location:** `src/daemon/runtime/maintenance/compact-api.ts:97` (and the doc comment lines 83-89), plus the test that pins it `tests/daemon/runtime/maintenance/compact-api.test.ts:148`.

```ts
// compact-api.ts:93-99
export const COMPACTABLE_KEY_COLUMNS = Object.freeze({
  skills: "id",
  rules: "key",
  entity_attributes: "claim_key",
  epistemic_assertions: "claim_key",   // ← WRONG
  dreaming_state: "id",
});
```

**Why it is wrong (independent verification against the real writer):** The `epistemic_assertions` version chain is written by `recordAssertion` in `src/daemon/runtime/ontology/control-plane.ts:482-484`:

```ts
await appendVersionBumped(storage, target, scope, {
  keyColumn: "id",
  keyValue: key,            // key = assertionKey() = `assert_<sha256(speaker:predicate:content)>`
  row: [ ["id", val.str(key)], ... ["claim_key", val.str(assertion.claimKey ?? "")], ... ],
});
```

The version chain is keyed by **`id`** — a STABLE deterministic hash (`assertionKey()`, control-plane.ts:445-450) that is shared across every version of the same assertion (a revised belief from the same speaker/predicate/content appends version N+1 under the same `id`). The `claim_key` column is `assertion.claimKey ?? ""` — an OPTIONAL cross-link to a claim that is **frequently empty** and is NOT the assertion's own lineage key. There is NO reader anywhere that resolves the current assertion by `claim_key ORDER BY version` (grep for `epistemic_assertions` + `claim_key` returns only the compaction map itself — no reader).

**Consequence (identical to the `skills` bug security fixed):** `compactVersionHistory` discovers keys via `SELECT DISTINCT claim_key` and resolves "highest version per key" on `claim_key`. For `epistemic_assertions` this is a **safety+no-op double-fault**:
- Every assertion with `claim_key=""` (the common case) collapses into a SINGLE bogus "chain" keyed by the empty string, so "highest version" resolves across DISTINCT logical assertions — the cross-entity-highest hazard the map's own doc-comment (lines 86-89) warns against.
- Assertions that DO carry a real `claim_key` share it with the `entity_attributes` claim and with each other, again mis-grouping.
- Net effect in production: compaction on `epistemic_assertions` either silently reaps nothing useful or groups unrelated rows — it does NOT bound that table's growth, which is the entire point of the PRD. This is an OPEN AC-6 criterion ("version-bumped tables ... reaped"), not DONE.

**Contrast — the other four are CORRECT (independently verified):**
- `skills → "id"` ✅ — `product/api.ts:151` resolves current skill via `buildHighestVersionSql("skills", "id", ...)`; there is no `key` column on skills. (This is the one security fixed.)
- `rules → "key"` ✅ — `product/api.ts:164` resolves via `buildHighestVersionSql("rules", "key", ...)`; `rules` has a real `key` column; `keyed-engine.ts:274` writes keyed by `"key"`.
- `entity_attributes → "claim_key"` ✅ — the supersede path keys the chain by `claim_key`: `supersede.ts` `readPriorActiveId` uses `buildHighestActiveVersionSql(claimKey)` (`knowledge-graph.ts:354` → `WHERE claim_key=… ORDER BY version DESC LIMIT 1`), and `readMaxClaimVersion` computes N+1 per `claim_key`. The `id` is unique-per-version (`attributeVersionId(aspectId, slot, version)`), so keying by `id` WOULD be a singleton no-op — the map is right here.
- `dreaming_state → "id"` ✅ — `trigger.ts:285` writes `appendVersionBumped({keyColumn: "id"})` with a deterministic per-scope `id` genuinely shared across the chain.

So the fix is right on 4 of 5 tables; `epistemic_assertions` was given `claim_key` by false analogy to `entity_attributes` (both are "claim tables"), but unlike `entity_attributes` the assertion writer never adopted the `claim_key`-keyed supersede path — it uses plain `id`-keyed `appendVersionBumped`.

**Recommended remediation (do not implement here — grade, not fix):** Change `epistemic_assertions: "claim_key"` → `epistemic_assertions: "id"` in `COMPACTABLE_KEY_COLUMNS` (compact-api.ts:97), correct the doc-comment (lines 83), and flip the unit assertion `compact-api.test.ts:148` to `expect(keyColumnFor("epistemic_assertions")).toBe("id")`. Add a guard test that pins each entry to its writer's `keyColumn` so this class of bug cannot regress silently. Re-run the live itest if a real `epistemic_assertions`-shaped chain can be seeded; otherwise the unit pin is the floor.
