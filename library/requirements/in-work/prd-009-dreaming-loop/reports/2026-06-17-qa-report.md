## Suggestions (consider improving)

### S-1 — Counter-wedge probe default is deliberately conservative but undocumented for operators

**File:** `src/daemon/runtime/dreaming/trigger.ts:207-208`

```ts
// Default probe: never report terminal — never enqueue a 2nd pass on a guess (FR-6).
this.pendingTerminal = deps.pendingTerminal ?? (() => Promise.resolve(false));
```

The default `PendingJobTerminalProbe` returns `false` forever when no real probe is injected (daemon-assembly deferred). Security-worker-bee flagged this as a Medium (counter-wedge: a crashed pass holds `pending_job_id` until assembly wires the real probe). The code comment explains the rationale. Suggest a `CONVENTIONS.md` or a `// TODO(PRD-009-assembly)` cross-reference at this line so the assembly step cannot miss it. No behavior change needed.

**Severity:** Suggestion. The plan explicitly defers assembly; security-worker-bee documented this; the conservative default is the correct safe posture.

### S-3 — `sqlIdent` applied twice to `tableName` in trigger constructor

**File:** `src/daemon/runtime/dreaming/trigger.ts:214-215`

```ts
this.tableName = sqlIdent(deps.tableName ?? DREAMING_STATE_TABLE);
// ...
private tbl(): string {
  return sqlIdent(this.tableName);
}
```

`this.tableName` is already the quoted/escaped output of `sqlIdent`. Calling `sqlIdent` again on it in `tbl()` double-quotes the identifier (e.g. `"\"dreaming_state\""`) for all SQL paths that use `tbl()`. The live test passes because the namespaced throwaway table name (`ci_dreaming_<run_id>`) contains only alphanumerics/underscores so double-quoting is harmless. In production with the default `DREAMING_STATE_TABLE` constant (which is `dreaming_state`, also safe), the behavior is functionally correct. However it is a latent correctness issue if a table name with special characters is ever passed. Note: the `latestRow` method uses the stored `this.tableName` inside a string literal directly (line 262: `` `SELECT ${cols} FROM "${this.tbl()}"` ``) — the outer `"` plus the inner `sqlIdent` result means the identifier ends up double-quoted. Current inputs are all alphanumeric so no functional defect today.

**Severity:** Suggestion. No test failure, no security risk, but worth cleaning up before assembly to avoid a latent correctness trap.

## Notes (plan ambiguity / documented deferrals)

**N-1 — Daemon-assembly deferred (consistent with plan):** The hand-off that wires `DreamingTrigger` into the maintenance loop, the `DreamingRunner` into the job queue handler, and the `DreamStateReader`/`DreamJobEnqueuer` into the CLI is explicitly deferred to an assembly step. The plan states this clearly ("daemon-assembly is DEFERRED — constructed-and-tested in Wave 1"). All seams are stable and tested. Not a gap.

**N-2 — Identity-file loader / DREAMING.md pending PRD-019 (consistent with plan):** The real identity preset and `DREAMING.md` file load require PRD-019. The default source yields empties + `DEFAULT_DREAMING_TASK_PROMPT`, which is correct and structurally enforces the dreaming-only invariant. Not a gap.

**N-3 — `--compact` CLI does not set `pending_job_id` on the trigger path (security Medium, plan-consistent):** The CLI `dream trigger --compact` enqueues via the injected `DreamJobEnqueuer` seam. Setting `pending_job_id` on the `dreaming_state` row requires the daemon assembly to wire the enqueuer to call the trigger's append-only path. Until assembly, the single-pending guard (a-AC-3) does not cover CLI-initiated jobs. Security-worker-bee documented this as Medium. The plan's scope table correctly says the CLI only exposes `--compact` "until daemon-assembly wires a force-trigger". Not a gap against the plan; tracked by security.

**N-4 — b-AC-4 framing (merge_entities routes via supersede_attribute):** The plan's b-AC-4 reads "prior rows are advanced in status on the append-only path". The test proves `supersede_attribute` routes via `claim.supersede` (a bounded direct-apply operation that uses the shared `supersede.ts` append-mark helper), leaving the original row intact and adding a superseded-mark version. `merge_entities` maps to `entity.merge` which routes to pending review (b-AC-2 guard), so its append-only lineage is preserved by the 008c control-plane path rather than the supersede helper directly. This is the correct interpretation of "prior rows remain on disk with lineage intact" for merge ops: they land in the proposal queue with full provenance. The ledger's framing ("009b Bee proved the append-only status-advance via supersede_attribute since merge_entities routes to pending") is accurate. Not a gap.
