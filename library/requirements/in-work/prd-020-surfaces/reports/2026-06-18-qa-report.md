## Suggestions (consider improving)

**S-1 (Suggestion)** — `src/commands/status.ts` imports `healOrgDrift` from `src/daemon/runtime/auth/device-flow.js` in `tests/commands/status.test.ts:22` for the a-AC-4 live heal test. This is an indirect daemon-module import in a test for a non-daemon root. The production `status.ts` handler itself is clean (all seam-injected, no storage import), but the test imports `healOrgDrift` and `saveCredentials` directly from `src/daemon/runtime/auth/` to prove the 011b heal works. This is not an invariant violation (the invariant scans source roots, not test files), but the pattern means a change to `device-flow.ts` can break a CLI test without a daemon-test signal. Consider extracting the `healOrgDrift`+`saveCredentials` round-trip into a dedicated `tests/daemon/runtime/auth/drift-integration.test.ts` so CLI tests remain seam-only. Non-blocking.

**S-2 (Suggestion)** — `src/cli/index.ts:32` constructs `createLoopbackDaemonClient()` but leaves the handler seams (`auth`, `health`, `drift`, `connector`, `dashboard`) unbound. The honest deferral message ("not wired in this build") is correct per D-7. However, there is no test that validates the "not wired" fallback message for each unbound seam (only the auth passthrough has a coverage path in `tests/commands/dispatch.test.ts`). A short test asserting that `setup`, `status`, `dashboard`, and `hook` all exit non-zero with the honest message when their seam is missing would lock in D-7 honesty and prevent a silent regression where a seam is wired without a real implementation. Non-blocking.

**S-3 (Suggestion)** — The F-2 Medium finding (all daemon read handlers trust `x-honeycomb-org` header for the partition with no token-claim cross-check) is documented as systemic and pre-existing. The security report recommends surfacing the validated `Identity` from `permissionMiddleware` into the Hono context so all handlers derive `{ org, workspace, actor }` from the token claim. The three 020 handlers (`mountDashboardApi`, `mountNotificationsApi`, `attachSessionsPrune`) are among the first that would benefit from this fix. Given the `PruneActorAuthority` seam already isolates the destructive path, this is not a blocker for 020, but it is the natural first wave of a PRD-021-level daemon-wide hardening ticket. Flag for the next planning cycle.

## F-2 / Medium Blocking Assessment

**Explicit call: F-2 does NOT block PRD-020 ship.**

F-2 (header-trusted org partition on daemon read handlers) is a systemic, daemon-wide pre-existing architecture pattern present since PRD-011. It is not introduced by PRD-020. The 020 read handlers (`/api/kpis`, `/api/diagnostics/sessions`, `/api/diagnostics/settings`, `/api/graph`, `/api/rules`, `/api/skills`) are all protected by `permissionMiddleware` in `team`/`hybrid` mode — a request without a valid token reaches none of them. The data they return is display-only (org name, port, session metadata, rule titles, skill names) and contains no credentials, tokens, or private keys. The local mode default is single-user with no second tenant to cross-read. The F-1 destructive path (the only new write endpoint in 020) was fixed with a fail-closed gate before this QA run. Shipping PRD-020 with F-2 documented as a PRD-021-class follow-up is the correct risk call.

The F-3 Low (silent coercion of malformed view data in `parseSnapshot`) is an intentional fail-soft design choice in the render path. It does not block ship.

## Key Design-Decision Verification

### D-4: No forked merge engine

- `src/notifications/auto-wiring.ts:38–48` delegates entirely to `deps.connector.install()` / `uninstall()`. No merge logic in the file.
- `harnesses/cursor/extension/extension.ts:117–122` delegates `wireHooks` to `deps.hooks.wire()`, which is the `connectorHookWiring(connector)` factory over a real 019a `HarnessConnector`.
- `tests/cursor-extension/extension.test.ts:80–112` uses a genuine `CursorTestConnector extends HarnessConnector` backed by `createFakeFs` — the 019a `FakeFs` — proving the extension exercises the real 019a connector rules rather than its own fork.
- `tests/notifications/auto-wiring.test.ts` uses `ClaudeCodeConnector` (a real 019a connector) and the real `createFakeFs`. No second merge engine exists.

### D-6: Single dashboard view implementation

- `harnesses/cursor/extension/render.ts:56–63` calls `rendered.views.map(renderBlock).join("")` where `rendered` comes from `renderDashboard(source)` — the canonical 020b function. No view builders are re-implemented here.
- `tests/cursor-extension/extension.test.ts:410–428` asserts `webviewHtml === renderDashboardHtml(canonical)` where `canonical = await renderDashboard(source)`. This is a structural identity check, not just title-string presence.

### D-2: Thin-client invariant

The `NON_DAEMON_ROOTS` list in `tests/daemon/storage/invariant.test.ts:43–57` now includes `src/commands`, `src/dashboard`, `src/notifications`, and `harnesses/cursor/extension` (added by Wave 1). The invariant test ran green (3/3) post-security-fix.

### D-7: Deferral honesty

Every deferred seam is:
1. Clearly documented in `src/cli/index.ts:13–22`, each stream's `CONVENTIONS.md`, and the ledger Watchdog section.
2. When unbound, prints an honest "not wired in this build" message (`dispatch.ts:127–130`) rather than silently returning 0 or throwing.
3. Nothing in the 020 implementation claims to be live-wired end-to-end against a running daemon, a real `vscode` host, or a production assembly — all are constructed-and-tested behind seams.

### Security fix (F-1 / a-AC-2)

The `PruneActorAuthority` seam at `src/daemon/runtime/sessions/prune.ts:119–140` is fail-closed. `denyUnboundActorAuthority.resolveAuthorizedActor` returns `null` unconditionally. The mode guard at `prune.ts:340–348` enforces the 403 path for any `mode !== "local"` request when authority returns null. The `a-AC-2 SEC local mode is UNCHANGED` regression test (prune.test.ts:206–218) proves the paired-delete AC itself is unaffected by the fix.
