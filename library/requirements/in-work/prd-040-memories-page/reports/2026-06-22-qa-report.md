## Suggestions (consider improving)

- [ ] **Re-read convergence predicate short-circuits on `version > 0`**, `src/dashboard/web/pages/memories.tsx:655`

  The edit re-read polls with `rereadUntil(id, (rec) => rec.content === content || rec.version > 0)`.
  Because a persisted memory almost always carries `version > 0`, the predicate is satisfied on the
  FIRST read, so the poll-until-convergence backstop (5 attempts × 150ms) rarely engages in practice.
  This does NOT violate AC-4 — state is always set from the daemon `getMemory` read, never the form
  input, and the divergent-content test proves it (line 296-310). But against the project-memory
  DeepLake eventual-consistency rule, the `|| rec.version > 0` clause weakens the convergence intent:
  a stale read that still has the OLD content but a non-zero version returns immediately. Consider
  tightening to converge on the new content/version delta (e.g. `rec.content === content` and/or
  `rec.version > priorVersion`) so the poll actually waits out a stale segment.

  ```ts
  const persisted = await rereadUntil(id, (rec) => rec.content === content || rec.version > 0);
  ```

- [ ] **`version` column relies on heal-on-first-bump; a never-bumped row reads `version: 0`**, `src/daemon/runtime/memories/reads.ts:117`

  `SELECT_COLS` includes `sqlIdent("version")`, but `version` is not in the base `memories` ColumnDef
  list (`src/daemon/storage/catalog/memories.ts`) — the comment notes it "heals onto the table on the
  first bump." A freshly added memory that has never been modified may have no `version` column value,
  surfacing as `version: 0` → the detail view's `version` MetaRow renders nothing (it gates on
  `record.version > 0`). This degrades gracefully and is consistent with the additive-optional design,
  so it is not a gap. Consider a brief UI affordance (e.g. "v1" default) so a brand-new memory's
  version is legible rather than absent, if product wants version always-visible.
