## Warnings (should fix)

- [ ] **`POST /api/documents` ingests the URL string, not the document body (echo fetcher wired in prod)**, `src/daemon/runtime/sources/registry.ts:358-374`, `src/daemon/runtime/sources/document-worker.ts:350-354,756-758`

  `buildSourcesApiDeps` constructs the real `DurableDocumentWorker` via `createDocumentWorker(...)` but passes **no** `fetcher`, so the worker falls back to `echoDocumentContentFetcher`, whose `fetch()` returns `{ content: submission.url }`. The full lifecycle (extract→chunk→embed→index→done) genuinely runs and persists active `document_chunk` rows, so e-AC-3's literal bar ("ingest through the wired worker → done → recallable chunks") is met — the live itest's "recallable" URL is itself the chunked content. But an end user calling `POST /api/documents` with a real URL gets the URL text chunked, not the document at that URL. The worker contract is fetcher-injectable (`DocumentWorkerDeps.fetcher`), so the remediation is to thread a real URL/file fetcher into `buildSourcesApiDeps`. Recommend either wiring a real fetcher or annotating the document surface as "URL-as-content placeholder until a fetcher lands" so the capability is not overstated.

  ```ts
  // registry.ts:361 — no fetcher passed → echoDocumentContentFetcher (content = url)
  const documentWorker = createDocumentWorker({ storage, scope, queue, embed, ...(resolveTable ...) });
  // document-worker.ts:351 — the default fetcher echoes the URL as the document body
  async fetch(submission) { return { content: submission.url, title: submission.url }; }
  ```

- [ ] **Memory-pipeline stage gates default OFF — "live" is wired+enableable, not on-by-default**, `src/daemon/runtime/pipeline/config.ts:146,109,111`, `src/daemon/runtime/pipeline/graph-persist.ts:433-436`

  The pipeline worker is constructed, started, and leases the 5 kinds unconditionally (`assemble.ts:1680-1681`), and capture enqueues the entry job (`capture-handler.ts:192`) — those are genuinely live. But the stage handlers gate on config flags that all default `false` (`enabled`, `graph.enabled`, `graph.extractionWritesEnabled`), so on a default daemon boot with no `agent.yaml` pipeline block, extraction/decision/controlled-write/graph-persist no-op and `inlineLinkMemory` (c-AC-1) never fires. This is the documented false-safe posture and is exactly analogous to the dreaming default-OFF that PRD-045d explicitly blessed; the deterministic chain test (`pipeline-worker.test.ts` `enabledConfig()`) and the token-gated live itest both exercise the ENABLED path and assert real INSERTs into `memories`/`memory_history`/`entities`/`entity_dependencies`/`memory_entity_mentions`. No AC requires on-by-default. Flagged only so the close-out narrative is read precisely: the engines are *reachable and enableable*, not *firing on a vanilla boot*. No fix required unless the product intent is on-by-default.
