## Suggestions (consider improving)

### S-1 — b-AC-5 test at ~1.67s in CI (near the 5000ms Vitest ceiling)

- **Where:** `tests/daemon/runtime/sources/document-worker.test.ts` — the `b-AC-5` case ("a document delete soft-deletes…") consistently runs at ~1.6–1.7s in the sources-only suite.
- **What:** The test builds enough chunks (via 10-char chunkSize forcing multiple windows over a long URL) that the `FakeArtifactStore.query` parse loop dominates wall time. Currently well under the 5000ms timeout, but if CI gets slower (e.g. Windows runner contention) it could flake.
- **Recommendation:** No action required now. If CI flakiness appears on this test, reduce the test's URL length or pre-seed a smaller fixture.

### S-2 — `api.test.ts` a-AC-2 case self-imports the lifecycle (dynamic import inside a test)

- **Where:** `tests/daemon/runtime/sources/api.test.ts:144` — the `DELETE /api/sources/:id` test does `const { createSourceLifecycle } = await import("../../../../src/daemon/runtime/sources/lifecycle.js")` inside the test body to set up state, because the API handler delegates to the lifecycle internally rather than exposing a way to inject pre-indexed state.
- **What:** The dynamic import pattern is valid but slightly unusual — it conflates test-setup concerns across two modules. Not incorrect, but the intent is harder to follow than a direct `buildLifecycle` helper call.
- **Recommendation:** Cosmetic only. Consider extracting a shared `buildLifecycleForApiTest` helper to `tests/helpers/` if this pattern recurs.

### S-3 — `sources-purge-live.itest.ts` does not assert `chunksPurged`; only `artifactsPurged >= 2`

- **Where:** `tests/integration/sources-purge-live.itest.ts:188` — `expect(outcome.artifactsPurged).toBeGreaterThanOrEqual(2)`. Chunk status after purge is not independently verified in the live itest.
- **What:** The unit `lifecycle.test.ts` a-AC-2 case DOES assert chunk status (via `DOCUMENT_CHUNK_TABLE` rows). The live itest covers the artifact table only. The chunk table is created (`TBL_CHUNKS`) and cleaned up, but `chunksPurged` is not asserted on the live outcome. Given the live-proven purge bug history (the copy-forward fix), a belt-and-suspenders live chunk assertion would add confidence.
- **Recommendation:** In a follow-up, add a `currentStatus(storage, TBL_CHUNKS, chunkId, scope)` assertion to the live itest, consistent with the artifact read-back.
