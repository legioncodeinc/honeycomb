## Suggestions (consider improving)

### S-1 — Catalog `memory` pattern label predates the wiki-summary write path

- **File:** `src/daemon/storage/catalog/sessions-summaries.ts:90`
- **Observation:** `MEMORY_COLUMNS` is declared with `pattern: "update-or-insert"` (inherited from the PRD-003c VFS use case). The wiki-summary worker and synthesis explicitly bypass this pattern label and call `selectBeforeInsert` / `buildInsert` directly — correctly, and in alignment with a-AC-6 / b-AC-4. The label therefore no longer describes how PRD-017 code writes to this table.
- **No runtime impact:** `pattern` is metadata only; the registry lookup (`patternFor`) is never called by PRD-017's code paths, so the label does not govern the actual write. `audit:sql` confirms no `updateOrInsertByKey` call is emitted for `memory` by the summaries module.
- **Recommendation:** When the next PRD touching this table lands, consider adding a comment or splitting the entry to acknowledge that `memory` now supports two write patterns (VFS upserts via `updateOrInsertByKey` vs. wiki-summary rows via `selectBeforeInsert`). Not blocking.

### S-2 — `SummaryStore.writePlaceholder` signature change not reflected in contract JSDoc

- **File:** `src/daemon/runtime/summaries/contracts.ts:299`
- **Observation:** The `SummaryStore.writePlaceholder` interface signature declares `writePlaceholder(path: string, author: string): Promise<void>`, and the implementation in `worker.ts:245` accepts both parameters. However, the JSDoc comment on the interface at line 298 says "Write an in-progress PLACEHOLDER row at the path" without mentioning the `author` parameter. Minor doc gap; does not affect correctness.
- **Recommendation:** Update the JSDoc to mention the `author` parameter. Not blocking.

## Live Fix Assessment

### Fix 1 — `description` column added to `MEMORY_COLUMNS`

**File:** `src/daemon/storage/catalog/sessions-summaries.ts:70-71`

The `rowValuesFor` builder in `worker.ts:220-240` writes a `description` field. Before this fix, `MEMORY_COLUMNS` did not include `description`, so `healMissingColumns` could not add it on a production table lacking it, causing every INSERT to fail with "column does not exist." The fix adds:

```typescript
{ name: "description", sql: "TEXT NOT NULL DEFAULT ''" },
```

This is the correct, minimal, additive repair. It is heal-compatible (the column has a `NOT NULL DEFAULT ''` — safe for existing rows on schema heal). The comment at line 68-71 documents the rationale including the in-progress placeholder dependency. Assessment: sound.

### Fix 2 — Placeholder-excluding SBI probe in `writeSummary` and `synthesis.ts:writeRow`

**Files:** `src/daemon/runtime/summaries/worker.ts:287-293`, `src/daemon/runtime/summaries/synthesis.ts:235-237`

The original probe was `SELECT path FROM "memory" WHERE path = <path>`. Because the backend's `DELETE` is unreliable, a stranded `'in progress'` placeholder at the same path would make the probe return `alreadyPresent=true` and silently drop the real summary. The fix adds `AND description != 'in progress'` to the probe:

```typescript
`WHERE ${pathCol} = ${sLiteral(row.path)} ` +
`AND ${descCol} != ${sLiteral(IN_PROGRESS_MARKER)} LIMIT 1`
```

This means: "a real summary exists at this path" — a stranded placeholder does not satisfy the condition and cannot block a fresh write. The `written` field now reflects `isOk(inserted)` (the actual INSERT result) rather than `!alreadyPresent` (which masked a failed INSERT as success). Both fixes are logically correct, probe-safe, and mirrored symmetrically in `synthesis.ts`. Assessment: sound.

## MEMORY.md Write-Once Limitation Assessment

The ledger documents (final watchdog entry): "MEMORY.md is SELECT-before-INSERT write-once per scope (the literal b-AC-4 reading) — a re-synthesis after NEW summaries land is a no-op, so MEMORY.md does not refresh across runs without a version-bump."

Assessment of blocking status:

1. **b-AC-4 literally says:** "Given an existing MEMORY.md or thread-head row, when synthesis re-runs, then it uses SELECT-before-INSERT rather than an in-place UPDATE." The implementation satisfies this exactly — it inserts once and does not UPDATE.

2. **The limitation is a product gap, not an AC violation.** b-AC-1 says "when synthesis runs, then a MEMORY.md is written." It does not require the index to be refreshed on every call after the first. The plan never specified a refresh mechanism; that is what the "carried follow-up" note refers to.

3. **The limitation is honestly documented** in the ledger (the final watchdog entry), not silently present. The follow-up path (version-bumped MEMORY.md, same pattern as `ontology/supersede.ts` and `graph-persist.ts`) is named.

4. **Practical impact:** In steady-state use — where new sessions accumulate summaries over time — a single-tenant MEMORY.md does not auto-update to include newly landed summaries after the first synthesis. Agents navigating via MEMORY.md see only the summaries that existed at first synthesis time. This is a real product gap but is bounded by the documented follow-up.

**Verdict: not blocking.** The literal ACs are satisfied. The limitation is documented. The follow-up path exists. A user who bumps into it has a clear path to resolution.
