## Warnings (should fix)

- [ ] **Production `mountGraph` seam is not asserted by the assemble suite**, `src/daemon/runtime/assemble.ts:317`, `661-668` / `tests/daemon/runtime/assemble.test.ts`

  `mountGraph` is declared **OPTIONAL** on `SeamFns` (`readonly mountGraph?: typeof mountGraphApi`) specifically so the pre-existing PRD-021/022 recording-fake `SeamFns` stay type-compatible without editing those out-of-scope suites. The consequence: `assembleSeams` fires `mountGraph` only `if (seams.mountGraph !== undefined)`, and the assemble test suite (37 tests) contains **no** reference to `graph`/`mountGraph` â€” so the "every seam fires exactly once, after construction" guarantee (a-AC-2) that protects the other twelve seams does **not** extend to this one. The `api.test.ts` proves `mountGraphApi` works in isolation, but nothing proves the **composition root actually fires it** in a production-shaped assembly. A future refactor could drop the seam from `defaultSeamFns` and every test would stay green.

  ```ts
  // assemble.ts:661
  if (seams.mountGraph !== undefined) {
      try { seams.mountGraph(daemon, { storage, defaultScope, workspaceDir }); }
      catch (err: unknown) { /* fail-soft */ }
  }
  ```

  Remediation: add one assemble-suite case asserting `defaultSeamFns.mountGraph` is fired once with the live storage + `defaultScope` + `workspaceDir` (a recording fake that includes `mountGraph`), closing the once-fired gap for parity with the other seams.

- [ ] **No end-to-end `honeycomb graph build` CLI verb is wired**, `src/cli/` (absent)

  The PRD-014 index "API changes" enumerates `honeycomb graph build|diff|history|init|pull` CLI verbs as part of the feature surface, and the task framed the gap as "`honeycomb graph build` returned daemon 501." This change closes the **daemon endpoint** (`POST /api/graph/build`), which is the load-bearing half â€” but no CLI command reaches it: `grep` of `src/cli/` finds `graph` only in `ontology.ts`/`dream.ts`, and the daemon-client VFS (`src/daemon-client/vfs/read.ts`) does not call `/api/graph/build`. So an operator typing `honeycomb graph build` still has no path to the now-live endpoint. This is a scope boundary, not a defect in the wiring delivered, but the PRD's CLI surface for 014 remains unmet and a reader expecting "the 501 a user hit is gone end-to-end from the CLI" would be surprised.

  Remediation: confirm with `library-worker-bee` whether the CLI verbs are in-scope for this closure or tracked separately; if separate, note the deferral in the PRD so the index's "API changes" line is not read as satisfied. (Defer interpretation to the plan author â€” see Notes in traceability AC-API.)

## Suggestions (consider improving)

- [ ] **`GET /api/graph` reports counts from a different identity than the freshest file it returns**, `src/daemon/runtime/codebase/api.ts:240-255`, `111-139`

  `loadFreshestLocalSnapshot` picks the most-recently-modified `.json` in `<baseDir>/snapshots/` by mtime, while `defaultGraphBaseDir` is keyed by `identity.repo`. Within one repo dir this is correct, but if multiple commits' snapshots coexist (`<commit>.json` per build, 014b), the GET returns the freshest-by-mtime regardless of the requested HEAD â€” which is fine for the current "latest build" contract but could mislead once 014d's per-commit query surface lands. Consider keying the GET read on the resolved `commit` when present, falling back to freshest-by-mtime only when the commit file is absent. Non-blocking; the plan does not require per-commit GET resolution here.
