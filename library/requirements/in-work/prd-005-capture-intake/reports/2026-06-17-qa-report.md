## Suggestions (consider improving)

### S-QA-1 — Daemon bootstrap wiring not yet present

**Observation.** `createCaptureHandler` and `createEmbedAttachment` are not referenced in `src/daemon/index.ts` or `src/daemon/runtime/server.ts`. The capture route does not self-register at daemon startup.

**Evidence.** `src/daemon/index.ts` and `src/daemon/runtime/server.ts` — no import of `capture-handler` or `embed-client`. The security report noted this explicitly ("wiring note — not a vulnerability").

**Why this is a Suggestion, not a Warning.** The plan explicitly scopes real harness shim wiring to a later PRD (PRD-019). All testing is done through `daemon.group` / `app.request()`, which is the correct integration level for this PRD. The modules are complete; the wiring is the next PRD's job.

**Recommendation.** Track the bootstrap wiring in the PRD-019 plan. Confirm the entry point for `createCaptureHandler({ storage, sessionsTarget, queue, embed }).register(daemon)` is covered there, including whether `createEmbedAttachment` or `noopEmbedAttachment` is passed depending on `HONEYCOMB_EMBEDDINGS`.

---

### S-QA-2 — Security RECOMMENDED follow-ups S-2/S-3/S-4 remain open

**Observation.** Three Medium findings from `security-worker-bee` are unresolved (RECOMMENDED, not fixed):

- **S-2:** `HONEYCOMB_EMBED_URL` validated only by env string; no scheme check before captured text is POSTed to it. `src/daemon/runtime/services/embed-client.ts:166,246`.
- **S-3:** `workspace` interpolated raw into the DeepLake URL path. `src/daemon/storage/transport.ts:83` (PRD-002a/004a surface).
- **S-4:** Caller-asserted tenancy headers (`x-honeycomb-org`/`workspace`) with no real auth policy yet. `src/daemon/runtime/middleware/permission.ts:54-75` (PRD-004 follow-up).

**Severity assessment.** None rises to a blocker for PRD-005 merge:

- S-2 requires env-write access to exploit (not request-controllable); default is loopback.
- S-3 is a PRD-002/004 surface, already present before this PRD, not introduced here.
- S-4 is a documented accepted risk (auth policy is a later PRD; explicitly called out in PRD-004a).

These are correctly deferred. Flag them to the auth-module PRD owner.

---

### S-QA-3 — `capture-gate.ts` not exported from `src/shared/index.ts` barrel

**Observation.** `src/shared/index.ts` exports constants from `constants.ts` but does not export `shouldCapture`, `runCaptureGuarded`, or `CAPTURE_ENTRYPOINTS` from `capture-gate.ts`. Harness shims in a later PRD will need to import these, and the natural path would be `@honeycomb/shared` or the barrel — or a direct import path.

**Evidence.** `src/shared/index.ts:1-11` — no `capture-gate` re-export.

**Why this is a Suggestion.** The plan says harness wiring is a later PRD, so no shim currently needs the barrel export. However, if the PRD-019 shims expect `import { shouldCapture } from "@honeycomb/shared"`, this barrel gap becomes a blocker at that time. Better to add the export now while the module is fresh.

**Recommendation.** Add `export { shouldCapture, runCaptureGuarded, CAPTURE_ENTRYPOINTS, CaptureGateEnv, CaptureGateContext, CaptureDecision } from "./capture-gate.js";` to `src/shared/index.ts` unless the shim wiring PRD will import the module directly.
