## Suggestions (consider improving)

- [ ] **`schema` reason is hard-wired to `ok` in production; only tests exercise `missing_table`**, `src/daemon/runtime/assemble.ts:1024-1026`, `src/daemon/runtime/health.ts:86-91`

  The plan scopes `schema` as best-effort and conservative ("never a false `missing_table`"), and the builder correctly supports `schemaMissingTable: true`. But the live `healthDetail` thunk never passes that flag — `buildHealthDetail({ status: healthBit, embeddingsEnabled })` omits it, so production always reports `schema: "ok"` even when a required `memory`/`sessions` table is genuinely missing on a fresh partition (one of the exact dogfood pains the PRD cites in its "Why"). This is plan-compliant (the PRD explicitly defers the always-on missing-table signal: "no cheap always-on signal at the health seam"), so it is not a gap. Consider a follow-up that wires a cheap cached known-missing-table signal into the thunk so the `schema` chip can actually surface `missing_table` in the field, closing the loop on the missing-arm degradation the PRD set out to make visible.

  ```ts
  // assemble.ts:1026 — schemaMissingTable is never threaded, so schema is always "ok" in prod:
  const healthDetail = (): HealthDetail => buildHealthDetail({ status: healthBit, embeddingsEnabled });
  ```
