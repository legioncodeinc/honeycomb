## Suggestions (consider improving)

- [ ] **F-2: sqlLike wildcard injection (Low, carried from security report)**, `src/daemon/runtime/pipeline/decision.ts` (search candidate paths)

  The security report flagged that user-supplied values used in `LIKE` filters pass through `sqlStr` (which escapes quotes) but do not escape `%` or `_` wildcards before they reach the query. An attacker who can influence search terms could craft a wildcard-heavy pattern causing unbounded scan cost or unexpected match breadth. This is Low severity because the search is read-only (no data mutation path) and the attack surface requires an authenticated agent session. Remediation: add a `sqlLike()` helper that escapes `%` and `_` in user input before wrapping in `sqlStr`, then use it on any LIKE operand.

- [ ] **F-3: batchLimit has no ceiling (Low, carried from security report)**, `src/daemon/runtime/pipeline/config.ts:~L60` (`DEFAULT_RETENTION_BATCH_LIMIT`)

  `ClampedInt` enforces a minimum of 1 but no maximum. A misconfigured or adversarially-supplied `batchLimit` value in the thousands could cause a single retention sweep to issue a very large number of DELETE statements, potentially overwhelming the DeepLake transport. This is Low severity because the config is operator-controlled (not user-supplied at runtime). Remediation: add an upper clamp (e.g. 1000) to `ClampedInt` for `batchLimit` in `RetentionConfigSchema`.

- [ ] **Daemon-assembly wiring not yet connected (Suggestion — documented deferred scope)**, `src/daemon/runtime/pipeline/handlers.ts` and `src/daemon/runtime/pipeline/extraction.ts:onResult`

  The `createExtractionHandler` `onResult` callback defaults to no-op, and no daemon startup module calls `createPipelineHandlers()` + `createStageWorker()` with real service dependencies. This is explicitly documented in `handlers.ts:1-18` as consistent with PRD-004/PRD-005 deferral to PRD-020. It is not a PRD-006 gap. Noting here so the orchestrator has visibility: the pipeline stages are independently testable and the full live test suite uses the real stage implementations, but end-to-end flow through a running daemon requires the PRD-020 assembly module to wire the handler map and start the worker.

### PRD-006d Live-Determinism Assessment

The 006d live-determinism fix is genuinely sound. The two original root causes were:

1. `updateOrInsertByKey` used in-place UPDATE semantics, which DeepLake coalesces/loses on segment flush — replaced with `appendVersionBumped` (entity entity rows) and `appendOnlyInsert` (dep/mention rows), both of which append new version rows that survive segment consolidation.
2. `storage.query` was called directly for deps table access, bypassing `withHeal`, which meant the `entities_deps` table was never created live — fixed by routing all graph table access through heal-aware helpers.

Poll-convergence (PROBE_POLLS=8) is sound: the backend can only under-report presence (stale segment not yet visible), never invent a row that was not appended. Re-reading by deterministic id up to 8 times converges toward durable truth. Seven consecutive clean live runs confirmed. This is not a paper-over.
