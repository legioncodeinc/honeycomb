## Warnings (should fix)

- [ ] **a-AC-3: `installed` is always `false` in production — the live page understates wiring**, `src/daemon/runtime/assemble.ts:1092-1094` (and the default at `:1029`, doc at `:172-175`)

  The endpoint derives `installed` from an injected `ReadonlySet<string>` and this is correct, tested,
  and honest. But the production set is built from `options.harnessTargets`, which the production entry
  never supplies: `runAssembledDaemon()` is invoked with no options (`src/daemon/index.ts:205`), so
  `harnessTargets ?? []` is empty and every harness reads `installed: false` on the live
  `GET /api/diagnostics/harnesses`. a-AC-3 requires `installed === true` "for a harness whose
  hooks/identity (harness-sync) targets are present." The criterion's deeper intent ("installed+0 turns"
  and "uninstalled+N turns" both representable) IS met at the unit level (`harness-api.test.ts:168-185`
  proves both), and the value is honest (zeroed, never fabricated), so this is not Critical. But in the
  live dogfood the entire "installed" column / matrix mark will render `—` even for genuinely-wired
  Claude Code, so the page does not yet answer "which harnesses are wired?" — the PRD's headline question.

  ```ts
  // assemble.ts:1092 — derived from options.harnessTargets, which defaults to []
  const installedHarnesses: ReadonlySet<string> = new Set(
      (options.harnessTargets ?? []).map((t) => t.name),
  );
  // index.ts:205 — production calls with no options → harnessTargets is []
  runAssembledDaemon().catch((err: unknown) => { … });
  ```

  Remediation: thread a real installed-presence source into `assembleDaemon` — either the harness-sync
  `HarnessTarget` set the installers register, or the cheap on-disk hooks/identity presence check OQ-1
  proposed — so a wired harness reads `installed: true`. The doc comment already flags this as the
  019/021c follow-up; this finding makes the user-visible consequence explicit so it is tracked, not lost.

## Suggestions (consider improving)

- [ ] **OpenClaw/Hermes/pi/Codex specifics are hand-asserted flags, not shim-derived statics**, `src/daemon/runtime/dashboard/harness-registry.ts:129-146`

  Cursor's `agents` and Claude Code's lifecycle map are computed from real shim statics
  (`CURSOR_HOST_CLI`, `CLAUDE_CODE_EVENT_MAP`), but `contractedTools`/`mcpRegistration`/`agentsMdContext`/
  `userVisibleLogin` are literal booleans in `HARNESS_SPECIFICS`, grounded only by the adjacent comment.
  This is allowed by D-5 (the descriptor is the single place a harness's specifics live) and each comment
  is accurate against its shim (verified: OpenClaw "NO PreToolUse", `HERMES_MCP_MENTION`, etc.). Consider a
  follow-up that derives these from a shim-declared capability flag (as Cursor's are) so a future shim
  change can't drift the descriptor without a test catching it. Non-blocking; the c-AC-4 guard test
  already pins the current values.

- [ ] **Per-harness live stream filters `/api/logs` by path substring — thin honest signal**, `src/dashboard/web/pages/harnesses.tsx:302-306`

  `filterRecordsForHarness` keeps a record iff its request path contains the harness name. This is the
  honest client-side fallback c-OQ-1 sanctioned (the request-log record carries no `agent` tag), and it
  never fabricates a line — but for most harnesses few request paths mention the harness, so the live
  panel will usually show "waiting for daemon activity…". c-AC-2 is satisfied (real stream, filtered, no
  new endpoint, no secret), so this is opt-in: a future `?harness=` server filter or a `sessions`-turn
  view (c-OQ-3) would make the stream richer.
