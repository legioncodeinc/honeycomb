## Suggestions (consider improving)

### SUGGESTION-1 — Flaky PRD-011 test surfaces under full parallel suite run

**File:** `tests/daemon/runtime/auth/api-keys.test.ts:262`

The test `d-AC-4 a revoked key is rejected next request; siblings keep working` fails in the full `npm run ci` parallel suite run (exit code 1 from the test runner) but passes consistently when run in isolation. This is a pre-existing PRD-011 issue — the test was green on `main` in isolation, and the failure only manifests under concurrent worker load. It is unrelated to any PRD-015 change and no PRD-015 file is involved. However, since `npm run ci` exits 1, the gate is technically broken for this branch even though PRD-015's own 65 tests are clean.

**Recommendation:** isolate the api-keys test's shared state (likely a scrypt worker pool or bcrypt timing dependency) so it is deterministic under parallel execution. Not a PRD-015 action — flag to the PRD-011 owner for a follow-up fix.

**Impact on ship decision:** does not block PRD-015 specifically, but `npm run ci` exit code is 1 on this machine due to the flaky test. Recommend confirming the CI run in a fresh environment (or isolating this test) before merge.

### SUGGESTION-2 — sessions-concat LIMIT (carried from security report LOW-1)

**File:** `src/daemon-client/vfs/read.ts:143-153` (`buildSessionsConcatSql`)

The sessions-concat tier has no `LIMIT` — a long-lived session path loads every turn row into client memory. The security report documented this as LOW-1 and explicitly did not alter it (product decision). Worth revisiting alongside the truncation-notice pattern used in `generateVirtualIndex`.

### SUGGESTION-3 — write-buffer no absolute pending-size cap (carried from security report LOW-2)

**File:** `src/daemon-client/vfs/write-buffer.ts:235-300` (`enqueue`/`doFlush`)

A flood of writes to many distinct paths (or a permanently-rejecting backend) could grow the pending map without bound. The security report documented this as LOW-2. Optional hardening: an absolute pending-size ceiling for back-pressure and a concurrency cap on the flush fan-out.

## Dispatch-Through-Daemon Thesis (a-AC-6 QA Analysis)

The thin-client invariant is verified from three independent angles:

1. **Structural (compile-time):** `tests/daemon/storage/invariant.test.ts:79-108` scans all files under `src/daemon-client/` with a deny-regex that exempts ONLY `daemon/storage/sql.ts`. Any stray import of `client`, `index`, `writes`, `heal`, `vector`, `result`, or `catalog/*` fails this test. Verified: the PRD-015 refinement to this test (the `sql.ts` exemption + comment-strip) is sound — reviewed against the security report's 14-pattern adversarial table. The exempted module is import-free and pure.

2. **Affirmative (allowlist):** `tests/daemon-client/vfs/dispatch-invariant.test.ts:39-57` applies a positive allowlist: every `daemon/` import in the VFS must match `daemon/storage/sql\.js | daemon/runtime/codebase/(query|contracts)\.js`. A malicious import evading the deny-regex still fails here. Belt-and-braces.

3. **Behavioral (seam recording):** Every `read.test.ts` and `write-buffer.test.ts` tier test asserts on `dispatch.calls` — storage is provably reached ONLY through the seam because the fake records everything and there is no other path. The graph tier is proved zero-network by `dispatch.calls` being empty when the graph tier serves the read.

Import evidence from `read.ts:19-21`: imports `handleGraphVfs` (from `../../daemon/runtime/codebase/query.js`), `Snapshot` type (from `../../daemon/runtime/codebase/contracts.js`), and `sqlIdent`/`sLiteral` (from `../../daemon/storage/sql.js`). No storage client import. Import evidence from `write-buffer.ts:48`: imports `eLiteral`/`sqlIdent`/`sLiteral` from `../../daemon/storage/sql.js`. No storage client import. Both confirmed by `npm run audit:sql` exiting 0 on 121 files including all 7 VFS files.

## Schema Reconciliation Verification (015b Bee note)

The 015b Bee noted it wrote only the existing `memory` table columns (`path`, `summary`, `summary_embedding`) and explicitly did not write the PRD-prose columns `size_bytes` and `description`, which do not exist in the live `memory` table. This is correct behavior, not a silent AC gap:

- `buildMemoryInsertSql` (`write-buffer.ts:463-466`) inserts `path`, `summary`, `summary_embedding` — the columns that exist.
- `buildMemoryUpdateSql` (`write-buffer.ts:470-478`) rewrites `summary` and `summary_embedding` only.
- The PRD-015b FR-5 text says "rewrites `summary`, `summary_embedding`, `mime_type`, `size_bytes`, and `last_update_date` (plus optional `project`, `description`)" — a description of intent from authoring time. The actual table schema, as established by PRD-001 through PRD-013, does not include `size_bytes` or `description` on the `memory` table; writing those columns would fail at runtime.

The reconciliation is correct. The ACs do not specify these columns; only the FR prose does. No AC gap.
