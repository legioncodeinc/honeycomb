## Suggestions (consider improving)

### S-1 — Barrel does not export `gatePhase` or `createShapingPhase`; only `authorizationPhase` reaches the barrel

**File:** `src/daemon/runtime/recall/index.ts:93-94`

```typescript
export { type GatePhase, type RecallHit, type RecallResult, noopGatePhase } from "./gate.js";
export { type ShapedCandidate, type ShapedPool, type ShapingPhase, noopShapingPhase } from "./shaping.js";
```

The authorization phase's concrete implementation (`authorizationPhase`) is re-exported at `index.ts:89`, but the equivalent concrete symbols for the gate (`gatePhase`) and shaping (`createShapingPhase`) and traversal (`makeGraphTraversalPhase`) are absent from the barrel. Tests import them directly from their module files, which works for tests. For the eventual daemon-assembly module that wires all five phases into `createRecallEngine`, it will need to import from at least four separate paths rather than one barrel. This is not a correctness issue — the engine injection seam is intentional and the tests prove it — but the asymmetry will produce confusion when assembly lands. Consider either exporting all concrete phase implementations from the barrel, or none (with a note that assembly imports from the module files directly).

**Severity:** Suggestion. No production behavior is affected today.

### S-3 — The `audit:sql` pre-built-constant gap and transport echo: two open security follow-ups from `security-worker-bee`

**File:** `library/requirements/in-work/prd-007-retrieval/reports/2026-06-17-security-report.md`

`security-worker-bee` flagged two Low items as follow-ups rather than blockers:

1. **`audit:sql` pre-built-constant gap**: the script's regex identifies escaping-helper usage by pattern but cannot distinguish a legitimate pre-built constant from a hand-written literal if the constant is built at declaration time outside any function body. The current 56-file scan shows clean results, but the gap means a pre-built constant injection could slip the scan. Adding a dedicated test that asserts every constant string used in SQL fragments routes through a helper would close this.

2. **Transport echo**: the token is never hardcoded or logged in application code, but the raw `TransportRequest` (carrying `sql`, `org`, `workspace`) is exposed on `FakeDeepLakeTransport.requests` in tests. In production code the real transport does not echo the request; this is a test-helper surface, not a prod path. Confirm that no integration test or harness writes `FakeDeepLakeTransport` to a persistent log.

Neither item is reopenable for the PRD-007 ship decision (both were assessed as Low by `security-worker-bee` post-fix). Recording them here so they are traceable to a future hardening issue.

**Severity:** Suggestion (inherited from security follow-up assessment). Recommend opening a tracking issue for item 1.

## Security Thesis Confirmation

The authorization boundary was independently audited by `security-worker-bee` (report: `reports/2026-06-17-security-report.md`) and confirmed here by code inspection:

1. **IDs-only invariant:** `Candidate` at `contracts.ts` carries `id`, `scores`, `provenance` — no content field. Every phase from collection through shaping asserts IDs only. The gate (`gate.ts:186+`) is the sole phase that issues a content `SELECT`, under the carried `pool.context.clause`.

2. **Single chokepoint:** Every memory query routes through `buildScopeClause` at `scope-clause.ts:182`. The authorization test at `authorization.test.ts:402-428` adversarially verifies injection attempts in the IN-list and tag filter are doubled-quoted inert, and the roster lookup escapes the policy group value.

3. **Fail-closed:** Blank/whitespace agent id → `isolated` + `ScopeClauseError`; unknown policy → same. `group` with no resolved peers → own-only (not an error). Never wider on any failure path.

4. **Partition beneath:** The outer ring (org/workspace in `QueryScope`) is enforced server-side in `transport.ts:83-89` (URL path + header). `c-AC-6` test confirms that even with a cross-workspace id in the IN-list, the partition blocks it — proven with the `authorizedByWorkspace` model.

5. **H-1/H-2 fixes confirmed:** `traversal.ts:162-196` — `buildProjectEntitySql` and `buildEntityFtsSql` both route through `sqlLike`; `audit:sql` passes clean.
