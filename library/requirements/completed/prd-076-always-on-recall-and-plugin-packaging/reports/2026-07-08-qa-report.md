## Warnings (should fix)

- [ ] **F-1: the registered MCP bundle does not physically ship inside the plugin tree** — `esbuild.config.mjs:301`, `harnesses/claude-code/.mcp.json:6`, `package.json` `files`

  The registration path is install-safe and correct: `${CLAUDE_PLUGIN_ROOT}/mcp/bundle/server.js` (env `{}`, `command: node`). But esbuild emits the MCP bundle **only** to repo-root `mcp/bundle`, `harnesses/claude-code/mcp/` does not exist, and the npm `files` allowlist lists `mcp/bundle` but not `harnesses/claude-code/mcp/bundle`. Claude Code installs the plugin from source `./harnesses/claude-code` and does not copy files outside the plugin root, so `${CLAUDE_PLUGIN_ROOT}/mcp/bundle/server.js` will not resolve from an installed plugin — the `memory_search`/`hivemind_search`/`hivemind_read`/`memory_store` tools would not appear in a real session.

  ```json
  // harnesses/claude-code/.mcp.json:6 — correct path, but the target isn't shipped there
  "args": ["${CLAUDE_PLUGIN_ROOT}/mcp/bundle/server.js"],
  ```

  **Ruling:** ACCEPTABLE FOLLOW-UP / RELEASE GATE. It fails no 076b sub-AC as written (all six met + tested), is outside 076b's file ownership, and does not break the build/tests or the other three PRD-076 arms. It IS a must-close gate for m-AC-7's end-to-end intent. Remediation (ci-release): (1) in `esbuild.config.mjs`, copy the built `mcp/bundle/server.js` (+ ESM `package.json` stamp) into `harnesses/claude-code/mcp/bundle/`; (2) add `harnesses/claude-code/mcp/bundle` to the `files` allowlist; (3) extend `pack:check`/a distribution test to assert the plugin-internal bundle is present. Note: 076b's `.mcp.json` and 076c's `skills/`/`commands/` already ship via the marketplace source dir (they live under the plugin root); only the repo-root `mcp/bundle` is affected.
