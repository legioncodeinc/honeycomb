## Warnings (should fix)

- [ ] **FR-4 hook-type stubs absent from claude-code and cursor harness entries**, `harnesses/claude-code/src/index.ts:1-10`, `harnesses/cursor/src/index.ts:1-10`

  PRD-001b FR-4 specifies that "the Claude Code bundle packs the `session-start`, `session-end`, `pre-tool-use`, `capture` hooks and the specialized background workers" and "the Cursor bundle packs `session-start`, `capture`, `pre-tool-use`, `session-end`, and `graph-on-stop`." Neither harness stub names or exports any of these hook types; both export only a single `activate()` that calls `bootHarness()`. The pi stub similarly exports only `activate()` despite FR-4 specifying `wiki-worker` and `skillify-worker`.

  The Non-Goals section of PRD-001b ("runtime behavior of the bundled code") creates genuine ambiguity about whether FR-4 requires the hook surface at the foundation layer or only at the later PRD. However the plan text is unambiguous that these hook types are part of what each respective bundle "packs," and the current stubs do not enumerate them. This is flagged as a Warning rather than Critical because: (a) the bundles compile and run; (b) the Non-Goals and the scope statement ("entry-point selection per target") partially excuse the deferral; (c) the missing stubs do not break any currently testable AC. Remediation: add skeleton exports (e.g. `export const hooks = ["session-start", "session-end", "pre-tool-use", "capture"] as const`) or clarify with `library-worker-bee` whether FR-4 hook enumeration belongs to PRD-001b or the later hook-integration PRD.

  ```typescript
  // harnesses/claude-code/src/index.ts — current state
  export function activate(): HarnessContext {
      return bootHarness("claude-code");
  }
  // FR-4 expects: session-start, session-end, pre-tool-use, capture, background workers
  ```

## Suggestions (consider improving)

- [ ] **`npm run ci` omits vitest; consider noting this is intentional**, `package.json:41`

  The `ci` script is `npm run typecheck && npm run dup`. The hivemind-v1 reference implementation includes `vitest run` in its CI gate. PRD-001 does not explicitly require tests, and none exist at this foundation layer (correct for a skeleton), but the absence of `vitest` from the `ci` script means there is no future-proofing guard to remind contributors to add tests before CI passes. Consider adding a comment in `package.json` explaining why no test runner is in the `ci` script at this stage, or adding a placeholder `vitest run` that will be a no-op until test files appear.

- [ ] **`cli.js` direct-run guard uses a fragile suffix match**, `src/cli/index.ts:23`

  The direct-run guard `process.argv[1]?.endsWith("cli.js")` catches the most common case but would also fire if a test runner or some other script were named `cli.js`. The cleaner form (as in the daemon-client pattern) is an exact `import.meta.url === \`file://${process.argv[1]}\`` check. Both forms are present as an `||` to ensure the guard fires when the bundle is invoked as `honeycomb`; the suffix fallback is not wrong, but it is less precise than the `import.meta` half of the disjunction.

  ```typescript
  // src/cli/index.ts:23
  if (import.meta.url === `file://${process.argv[1]}` || process.argv[1]?.endsWith("cli.js")) {
  ```

- [ ] **`sync-versions.mjs` outputs to `console.error` in script mode; `console.log` in sync call**, `scripts/sync-versions.mjs:59,113`

  The exported `syncVersions()` function accepts a `log` callback defaulting to `console.error`, so all sync output goes to stderr. When invoked directly as a prebuild script, the log lines in the terminal (`sync-versions: 0 written, 6 unchanged`) appear on stderr, which is correct for build-script tooling. However a future caller that pipes stdout to parse build output could be confused if they expect sync status on stdout. A brief JSDoc note on the `log` parameter clarifying the stderr-by-default choice would prevent accidental mismatch.
