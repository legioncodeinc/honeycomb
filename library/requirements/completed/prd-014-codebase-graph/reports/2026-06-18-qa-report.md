## Five-Axis Evaluation

### Axis 1 — Completeness

All 28 ACs are covered by at least one named Vitest test in a dedicated test file. No AC has only a tautology (e.g. `expect(true).toBe(true)`) or a stub. Key load-bearing assertions verified:

- **Determinism (index AC-1 / b-AC-2):** `snapshot.test.ts:237-253` builds the SAME files under two different `repoRoot` paths (`/worktree/alpha` vs `/totally/other/path/beta`) and two different `cacheBaseDir` temp directories, then asserts `finA.sha256 === finB.sha256`. The volatile observation mutation test at line 256-264 additionally confirms that mutating `observation.generatedAt`, `observation.worktreePath`, and `observation.nodeCount` — plus individual node `startLine` and `fanIn` — leaves `computeSnapshotSha256` unchanged.

- **Drift-refuse no-clobber (c-AC-1):** `push-pull.test.ts:265` has `expect(store.insertCount(), "drift must not overwrite the stored row").toBe(0)` as an explicit named assertion. The stored row is also checked to still carry the original hash after the drift outcome.

- **Pull hash-revalidation (c-AC-2):** The `realSnapshot()` helper builds a genuine finalized snapshot through the actual `buildAggregateSnapshot` + `finalizeSnapshot` pipeline. A tampered payload (a node name mutated while the claimed sha remains the original) is placed in the fake store, and the test asserts `outcome.kind === "refused"` with `reason === "hash-mismatch"`. This is not a mock of the hash function — it exercises the real `computeSnapshotSha256` round-trip.

- **Zero-network (d-AC-5):** The structural assertion at `query.test.ts:268-279` reads the actual `query.ts` source file and asserts the complete import list is `["./contracts.js"]` and that `fetch(`, `daemon/storage`, `node:fs`, `node:net`, `node:https`, and `DeeplakeApi` are absent from the source text. This cannot pass for a fake-compliant implementation.

## Security Report Hygiene Items — Acknowledged

The security report identified two Low findings. They are confirmed noted here (not remediated — both were classified hygiene-only by security-worker-bee with no code change required):

- **Low-1** (`realpath` canonicalization): the manual walk does not canonicalize `repoRoot` itself before walking. `repoRoot` is daemon-supplied and not attacker-controlled. Noted for defense-in-depth only.
- **Low-2** (cache eviction): the per-file content-addressed cache under `~/.honeycomb/graphs/<repo-key>/.cache/` has no eviction policy. Entries accumulate over time proportional to unique file contents. No storage bound enforced. Noted as a future operational concern; `MAX_DISCOVERED_FILES` bounds the build, not the cache.

## Suggestions (non-blocking)

**S-1** (Suggestion): The `b-AC-6` atomic-write crash simulation (`snapshot.test.ts:353-376`) is thorough but the "crash" is simulated by writing a garbage temp file and never renaming it — it does not simulate a crash that leaves a partially-written FINAL file (the scenario the atomic write is designed to prevent). The test correctly proves the prior file is untouched when the rename does not happen; a complementary test that confirms `JSON.parse` succeeds on the final file after a successful write is present. No gap here — the test is valid — but a future hardening pass could add a test that truncates the FINAL file path mid-write to prove the rename-based atomicity against a more adversarial scenario. This is advisory only.

**S-2** (Suggestion): The seven structural language extractors (`structural.ts`) use a coarse export/visibility heuristic (`isExported`): Go/Java use capitalization; Rust uses `visibility_modifier`; Python/Ruby/C/C++ use leading-underscore-as-private. The plan's open question ("Should non-default languages (beyond the nine) be pluggable?") and the coarseness of the C/Python heuristic are consistent with the plan's "lighter but real, not a stub" language and are acceptable for Wave 1. If `isEntrypoint` accuracy for Python/C symbols matters in a future iteration, a per-language refinement hook could be added to the `LanguageSpec`. No AC violation.
