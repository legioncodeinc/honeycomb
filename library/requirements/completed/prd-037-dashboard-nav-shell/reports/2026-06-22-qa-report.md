## Suggestions (consider improving)

- [ ] **Auto-collapse is one-directional — it never auto-expands when the viewport widens again**, `src/dashboard/web/app.tsx:123-133`

  The narrow-breakpoint effect only sets `collapsed` to `true` (`apply` does nothing when `matches` is false), so a user who narrows then re-widens the window stays collapsed until they manually toggle. This satisfies 037a AC-6 as written (collapse under a narrow breakpoint; the manual toggle still works) and is a defensible "respect the user's last intent" choice, but a one-line symmetric `setCollapsed(matches)` — gated on whether the user has manually overridden — would feel more responsive. Optional; the plan neither required nor prohibited auto-expand.

- [ ] **The `dynamic`/`DynamicGroup` registry contract is defined and unit-tested but not yet rendered by the sidebar**, `src/dashboard/web/registry.tsx:65-68`, `src/dashboard/web/sidebar.tsx:200-203`

  037c AC-5 is satisfied: the `DynamicGroup.resolve` contract exists and a test proves a resolver returns N sub-items the sidebar *can* render (and `[]` yields none). But `Sidebar` currently maps only top-level `entries` to `NavItem`s and does not yet walk `entry.dynamic.resolve(...)` to render child rows in the live tree — by design, since the live data source is PRD-039's call (parent OQ-3) and 037c ships only the contract. Worth a one-line forward-reference comment at the `entries.map` site noting that dynamic child rendering lands with PRD-039, so a future reader does not mistake the absence for a gap.
