## Suggestions (consider improving)

### S-1 ‚Äî `exec.test.ts` b-AC-2 chunk-boundary tests are pure-unit, not spawn-backed

**Location:** `tests/daemon/runtime/secrets/exec.test.ts:146-167`

The two RollingRedactor chunk-boundary tests (`"catches a value split across chunk boundaries"` and `"catches a value split into two arbitrary halves"`) feed the redactor directly, bypassing the spawn path. They correctly prove the redactor's correctness, but they do not prove that `child.stdout.on("data")` chunks actually land in the redactor in the split-value scenario during a live spawn. The existing spawn-backed test at line 82-106 covers the common case. This is a depth-of-coverage note, not a gap in any AC.

**Recommendation:** Consider adding a spawn-backed variant where the script emits the secret in two `process.stdout.write` calls separated by a `setImmediate` to force two distinct `data` events, confirming the full path. Low priority ‚Äî the current coverage satisfies b-AC-2 per the plan.

### S-2 ‚Äî `api.test.ts` exec-wired tests reuse the same Hono/store fixture across describe blocks

**Location:** `tests/daemon/runtime/secrets/api.test.ts:163-178` (`buildWithExec`)

`buildWithExec()` is defined outside a `beforeEach` and called inline inside each test. For the current test count this is fine, but as the exec test surface grows the shared store state (secrets seeded inside individual tests) can interact if describe ordering shifts. Mirrors a pattern the rest of the test suite avoids (store construction inside `beforeEach`). No correctness issue today.

**Recommendation:** Move `buildWithExec()` into the `beforeEach` of a describe block that scopes all exec-wired tests together, matching the store-construction pattern in the rest of the secrets tests.

## Additional checks

### No-value-leak thesis ‚Äî end-to-end corroboration (QA angle)

Every route in `api.ts` was traced against the plan's FR-6 / a-AC-2 / a-AC-5 requirements:

- `GET /api/secrets` (`api.ts:164`) ‚Üí `store.listSecretNames` ‚Üí `{ names }`. Names only. Test: `api.test.ts:54-66`.
- `POST /api/secrets/:name` (`api.ts:172`) ‚Üí `store.setSecret`, response `{ ok, name }`, 201. Value consumed by the store, never in the response body. Test: `api.test.ts:68-78`.
- `DELETE /api/secrets/:name` (`api.ts:192`) ‚Üí `{ ok, name }`. Test: `api.test.ts:112-120`.
- `POST /exec` (`api.ts:111`) ‚Üí 202 `{ jobId, status }`. No value in the response. Test: `api.test.ts:179-213`.
- `GET /exec/:jobId` (`api.ts:131`) ‚Üí `ExecJobView` whose `stdout`/`stderr` are `RollingRedactor.current()` ‚Äî redacted by construction. Test: `api.test.ts:179-213`.
- `bitwarden/*`, `1password/*` (`api.ts:147-161`) ‚Üí 400 "use_exec" guidance when runner wired, 501 when not. Never a value. Test: `api.test.ts:241-249`.
- **There is NO `GET /api/secrets/:name` route** ‚Äî confirmed in `api.ts` (no such registration). Absence asserted by test `api.test.ts:82-94` (404).

`getSecretValue` is called in `store.ts:321` (the store itself), in `store.ts:441` (the `createSecretResolver` wrapper), and in `exec.ts:516` (the runner's `runJob`). No API handler calls it. Confirmed by grep: no call site in `api.ts`.

### Security HIGH-1 fix integration

`inheritableEnv()` (`exec.ts:672-678`) starts from `process.env` and filters out every key for which `isSensitiveEnvName(k)` returns true (`exec.ts:658-661`). The exact set includes `HONEYCOMB_DEEPLAKE_TOKEN` and any name matching `TOKEN`/`SECRET`/`API_KEY`/`APIKEY`/`PASSWORD`/`PASSWD`/`CREDENTIAL`/`PRIVATE_KEY` case-insensitively. Resolved job secrets are then layered on top at `exec.ts:524` and `exec.ts:539` ‚Äî after the strip ‚Äî so a job secret named `MY_API_KEY` is injected correctly and enters the `values` redaction set. The two regression tests confirm:

1. `HONEYCOMB_DEEPLAKE_TOKEN` and `SOME_PROVIDER_API_KEY` are absent from the child's `process.env` dump (stripped before child sees them, not merely redacted). `PATH` is present (`exec.test.ts:344-372`).
2. A job secret named `JOB_API_KEY` is used by the child (`view.stdout` contains `"used:yes"`) and is redacted in the output view (`view.stdout` does not contain the raw value) (`exec.test.ts:375-395`).

Both assertions are meaningful: they run a real `child_process.spawn` and inspect real output.

The 501 stubs confirm premature calls fail loud. No AC required wiring in this cycle; the plan correctly scoped it as deferred.

The `src/daemon/runtime/secrets/` tree contains exactly the six files the plan called for. `tests/daemon/storage/invariant.test.ts` (existing) enforces that the secrets module opens no DeepLake handles from a non-daemon root. `audit:sql` confirms zero SQL in the secrets tree.
