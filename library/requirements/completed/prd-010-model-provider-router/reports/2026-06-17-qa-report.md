## Suggestions (consider improving)

- [ ] **d-AC-3 pin test: router-side honour of pin is seam-level only**, `tests/cli/route.test.ts:302-313`

  The AC states "routes to the pinned target until unpinned." The test exercises `RoutePinStore.resolve` on the fake seam directly (`store.resolve("memory_extraction", SCOPE)`) and confirms the pin is stored. This is the correct CLI-boundary posture — the CLI is a thin client and the router's internal honour of pins is a daemon-assembly concern. However, the test comment at line 302 says "simulated by store.resolve" rather than stating that router-side enforcement is verified in a separate integration concern (the router does NOT yet read the RoutePinStore at routing time in this branch — that is part of the deferred daemon-assembly wiring). The seam-level assertion is sufficient for the CLI's AC, but a future reader could mistake it for proof that the router actually short-circuits mode selection on a pin. Consider adding a one-line comment at that test site clarifying: "The router's honouring of pins is a daemon-assembly-step concern (the pin-store endpoints are deferred; this test verifies the CLI sets the pin correctly via its seam)."

- [ ] **`RoutingExhaustedError` error message exposes workload name to callers**, `src/daemon/runtime/inference/router.ts:562`

  ```ts
  super(`inference: routing chain exhausted for workload "${decision.workload}" (no target served)`);
  ```

  The workload name is low-sensitivity routing metadata (it is also recorded to the telemetry table). However, `gateway.ts:errorResponse` only redacts `ProviderError` instances; a `RoutingExhaustedError` falls through to the generic `internal routing error` path (confirmed: `redactedError` returns 500/"internal routing error" for non-ProviderError). So the workload name does NOT reach the HTTP client — the redaction holds. The error message in the exception object is visible to daemon logs if any logger is ever added, but none currently exists (confirmed by the security audit). Low-risk, but worth noting so the "zero logging in the module" posture is preserved when a logger is introduced.

- [ ] **FR-10 (request scoped to calling org/workspace/agent_id) is inherited but untested end-to-end**, `src/daemon/runtime/gateway.ts:533-538`

  The PRD requires "requests MUST be scoped to the calling org/workspace and `agent_id`". The gateway inherits the permission middleware (confirmed by security audit and gateway tests checking the 400 on missing org header at `gateway.test.ts:467-476`). The history endpoint validates the org header. However no test verifies that a request with org `A` cannot read history rows written by org `B` — that cross-org isolation is ultimately a DeepLake partition guarantee, not a handler guarantee. This is outside the test boundary (no live DeepLake in unit tests) and is correctly deferred to the live integration tests. A comment at `historyScope` would make this explicit.

## Deferred Assembly Wiring (D-9 / CONVENTIONS.md)

The following four items are explicitly deferred in `src/daemon/runtime/inference/CONVENTIONS.md:149-159` and the EXECUTION_LEDGER, matching the identical deferred-assembly pattern used by PRD-008 (ontology CLI) and PRD-009 (dreaming CLI):

1. **RouterModelClient swap**: daemon assembly replaces `noopModelClient` with `new RouterModelClient(router)`. Documented TODO at the assembly site (`router.ts:668-672`). The `RouterModelClient` class is fully implemented and tested in `router.harness.test.ts:153-166`.
2. **Gateway mount on real server**: `mountInferenceGateway` called on the real daemon's `/api/inference` + `/v1` groups. The function is fully implemented and tested via `daemon.app.request(...)` in `gateway.test.ts`. The route groups are pre-scaffolded in `server.ts` (`protect: true`).
3. **CLI bin registration**: `honeycomb route` entry-point registered in the CLI bin. The `routeMain`/`parseRouteArgs`/`runRouteCommand` functions are fully implemented and tested in `route.test.ts`.
4. **Pin-store endpoints**: `POST/DELETE/GET /api/inference/pins` endpoints. The `RoutePinStore` seam is defined in `route.ts:161-167`; the router reading the pin map before mode selection is a follow-on assembly step. The CLI correctly documents this via the seam comment at `route.ts:35-40`.

None of these deferred items affect the 22 AC module-behavior guarantees.

## Secret-Never-Leaks Thesis (QA Corroboration)

The security report established this by type-construction analysis. From the QA angle, tracing the end-to-end path:

1. **Config parse** (`a-AC-4`): `SecretRef` zod schema rejects any non-`${NAME}` value at parse time; the rejection message does not echo the rejected value (`config.ts:75-77`).
2. **Dump** (`a-AC-2`): `dumpInferenceConfig` emits `apiKeyRef` (the reference). No resolved key exists in `InferenceConfig` — there is no raw-key field by construction.
3. **Routing engine** (`b-AC-5`): `apiKey` materializes only as a local `const` in `executeWithFallback` (`router.ts:392`) for the duration of one provider call. Never written to `Target`, `RoutingDecision`, or `AttemptRecord`.
4. **Gateway error** (`c-AC-5`): `redactedError` replaces the thrown message with a fixed status-class string. The gateway test plants `sk-LIVE-SECRET-abc123` in a `ProviderError` and asserts it never appears in the response (`gateway.test.ts:384-402`).
5. **History store** (`c-AC-6`/`d-AC-5`): `toPersistedEvent` (`history-store.ts:77-86`) copies only `request_id/workload/serving_target/mode/attempts/blocked_candidates`. `RedactedRoutingEvent` has no field for key/prompt/completion.
6. **CLI status output** (`d-AC-5`): `RouteHistoryEvent` (mirrors `RedactedRoutingEvent`) has no key/body field. Test at `route.test.ts:448-541` asserts absence of `sk-`, `apiKey`, `apiKeyRef`, `messages`, `content`, `completion`, `prompt`.
7. **Live itest**: `routing-history-live.itest.ts:122-178` writes a row containing metadata only, asserts the on-disk `event` JSONB contains neither `SECRET_VALUE` (`"sk-live-must-never-persist"`) nor `REQUEST_BODY`, and checks the `sk-` regex on the raw SQL read-back.

The thesis holds at every layer by construction.
