## Suggestions (consider improving) — escalated

### C-1 — The PRD-011 stub token issuer is still wired into the `honeycomb status` org-drift healer (D-5 spirit)

- **Where:** `src/cli/runtime.ts:234` (`buildOrgDriftHealer` constructs `buildRealTokenIssuer()`), feeding `src/commands/status.ts:116-117` (`deps.drift.heal()` runs on every `honeycomb status`).
- **What:** `buildRealTokenIssuer` (`src/cli/token-issuer.ts:165-169`) returns the **local stub issuer** in the default (no `HONEYCOMB_AUTH_URL`) case, whose `reMint` mints a token via `encodeStubToken` (`token-issuer.ts:59-62`). `healOrgDrift` (`src/daemon/runtime/auth/device-flow.ts:206-236`) persists a re-minted token through `saveCredentials` into the SHARED `~/.deeplake/credentials.json`.
- **Why it matters:** PRD-023 D-5 says "no dead stub left wired into a prod path," and D-1 makes `~/.deeplake/credentials.json` a file Hivemind also reads. In the **normal** PRD-023 flow this code is inert: `activeOrg = creds.orgId` equals the token's org claim, so `healOrgDrift` returns `aligned` with no write (`device-flow.ts:221`). **But** if `HONEYCOMB_ORG_ID` is set to a different org than the credential, the drift healer would call the **stub** `reMint` and overwrite the shared real-DeepLake credential with a locally-minted stub token — silently breaking both Honeycomb and Hivemind connect. PRD-023 migrated the `org`/`workspace`/`whoami` verbs onto the real `DeeplakeAuthClient` but left this one drift-healer seam on the old stub.
- **Severity rationale:** Suggestion, not Warning/Critical — not exercised by any of the 9 ACs, gated behind a non-default env override, and pre-existing (PRD-021b wiring, not introduced by PRD-023). Security correctly did not flag it (it is a correctness/consistency risk, not a token leak).
- **Recommended remediation (NOT done in-place — out of small/safe scope):** rewire `buildOrgDriftHealer` to re-mint through the real `DeeplakeAuthClient.reMint` (or disable drift-heal when the credential's `apiUrl` is the real `api.deeplake.ai`), and add an AC-style test asserting `honeycomb status` with a mismatched `HONEYCOMB_ORG_ID` does NOT clobber a real `~/.deeplake` credential with a stub token. This touches the 021b seam and needs its own test, so it is escalated rather than patched here.
- **Ask:** decide whether to (a) migrate the drift healer onto the real client now, or (b) accept it as a documented local-single-user-only path and guard it from writing when `apiUrl === https://api.deeplake.ai`.

### C-2 — L-1: the DeepLake API endpoint is not https-pinned (confirmed Low, defer)

- **Where:** `src/daemon/runtime/auth/deeplake-issuer.ts:160-164` (`resolveApiUrl` only strips trailing slashes); `src/daemon/storage/config.ts:48` (`endpoint: z.string().url()` accepts `http://`).
- **What:** The default is `https://api.deeplake.ai` (`credentials-store.ts:246`) and the browser-open path is hard-pinned to https (`validateVerificationUrl` / `defaultBrowserOpener`, `deeplake-issuer.ts:351,381`). However, the `apiUrl`/`endpoint` itself — sourced from `HONEYCOMB_DEEPLAKE_ENDPOINT` or a file's `apiUrl` — is not scheme-pinned, so a deliberate operator override or a tampered file could route the bearer token over plaintext `http://`.
- **Disposition (matches security's L-1):** **Confirmed Low, defer.** The daemon is loopback/local-mode-first today, the default is https, and the http path requires deliberate action. Pinning `resolveApiUrl`/the zod `endpoint` to https risks breaking local-dev `http://localhost` test endpoints, so it is NOT a small/safe in-place fix; it belongs to the team-mode/non-loopback GA work as a documented follow-up. **Not remediated here.**
- **Ask:** track as a GA hardening item — add an https-pin to the endpoint resolution (with a documented `http://localhost` dev exemption) before non-loopback/team-mode ships.

### S-1 — Stale doc comment references the old credentials path (cosmetic)

- **Where:** `src/cli/runtime.ts:21` — the module doc says the loopback client stamps headers from `~/.honeycomb/credentials.json`, but the code calls `loadCredentials()` (`runtime.ts:268`), which now reads `~/.deeplake` first (legacy fallback). Behavior is correct; only the comment is stale.
- **Severity rationale:** Suggestion (doc-only, no behavior impact). Not worth a regression test, so left as a note rather than an in-place edit per the no-assertion-weakening / minimal-touch discipline.
