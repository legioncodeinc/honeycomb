## Warnings (should fix)

### W-1 — Path-filter `LIKE` escape is a no-op (missing `ESCAPE '\'` clause)

- **Where:** `src/daemon/runtime/logs/log-store.ts:414-415` (query) + `:515-519` (`escapeLikePrefix`).
- **What:** `escapeLikePrefix` backslash-escapes `\`, `%`, `_`:
  ```ts
  return value.replace(/[\\%_]/g, "\\$&");
  ```
  but the query binds the value into a bare `LIKE ?` with **no `ESCAPE '\'` declaration**:
  ```ts
  clauses.push(`${pathCol} LIKE ?`);
  params.push(`${escapeLikePrefix(query.path)}%`);
  ```
  In SQLite, with no `ESCAPE` clause, `\` is **not** an escape character — so the injected `\` becomes a literal pattern char and `_`/`%` stay live wildcards. The escape is dead code.
- **Proof (reproduced against `node:sqlite`):** a `?path=/api/memories_` filter (literal underscore) returns `/api/memories/recall`, `/api/memoriesZx/y`, AND `/api/memories_x/y` — instead of only the literal `/api/memories_x/y`. The `_` matched any character.
- **Impact:** a path filter containing a literal `_` or `%` over-matches. Request paths are `/api/...` segments that effectively never contain those characters, so the common case (`/api/memories`, `/api/memories/recall`) works; the comment's stated intent ("filters that literal underscore, not 'any character'") is simply not delivered. **Not a security issue** — security-worker-bee correctly verified the value is still a bound `?` (no injection); this is a pure correctness gap in an edge case.
- **Recommended remediation:** declare the escape char — `... LIKE ? ESCAPE '\'` — so `escapeLikePrefix` actually neutralizes the wildcards. (Plan FR-4/043a-AC-2 / 043b-FR-2 specify "exact/prefix" path filtering; this restores the documented prefix semantics.) Add one test row with a literal `_` in the path to lock it.

Flagging only so the "0 events" in the Turn detail is understood as intentional, not a bug.
