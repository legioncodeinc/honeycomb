## Warnings (should fix)

### W-1 — `enable` action has no UI affordance and writes an empty body if reached (a-AC-6 / b-AC-4 partial)

- **Location:** `src/daemon/runtime/dashboard/sync-api.ts:544-549` (`enable`) + `src/dashboard/web/pages/sync.tsx:107-160` (`RowControls`) + `src/dashboard/web/pages/sync.tsx:368-394` (`onAction`).
- **Evidence:** The engine's `enable` writes `req.native ?? ""` to disk:
  ```ts
  // sync-api.ts:546
  const path = await installTarget.write(req.assetType, "project", req.name, req.native ?? "");
  ```
  But `RowControls` renders **only** Promote / Pull / Demote / Disable — there is **no Enable button** (grep of `sync.tsx` finds `enable` only in the union type and comments, never as a rendered `<Button>`). And `onAction` (sync.tsx:374-381) sends only `{ assetType, name, honeycombId }`, never `native`. So `enable` is daemon-reachable but UI-unreachable, and if it were wired the page would install an empty artifact. The header comment at sync.tsx:145-146 ("a shared-but-not-local row can be enabled") describes behavior the UI does not provide.
- **Impact:** The "enable" leg of a-AC-6/b-AC-4's enable/disable pair is engine-only. Disable is fully wired, tested (`sync-api.test.ts:233-248`), and reflected; enable is not exercised end-to-end and would mis-write if it were. OQ-2 resolved enable/disable as a local install toggle — a disable with no re-enable path leaves a disabled asset unrecoverable from the page.
- **Recommendation:** Either (a) render an Enable control for a shared-but-not-local row and have the daemon re-read the on-disk/substrate native (mirroring how `promote` reads via `installTarget.read`) rather than trusting a page-sent body, or (b) explicitly scope enable out of v1 in the PRD and drop the engine method + comment to match. Do not ship the empty-body `?? ""` write path as reachable.

### W-2 — Activity "live follow" is a `/api/logs` poll, not the `/api/logs/stream` SSE subscription c-AC-2 specifies (c-AC-2 partial)

- **Location:** `src/dashboard/web/pages/sync.tsx:355-358` + `src/dashboard/web/wire.ts:817-820` (`logs`).
- **Evidence:** c-AC-2 reads: "New sync events stream into the feed via `/api/logs/stream` (SSE)… backfills recent records then follows the tail." The page instead re-reads the JSON snapshot on a 2.5s timer:
  ```ts
  // sync.tsx:355
  usePoll(async () => {
      const records = await wire.logs(MAX_ACTIVITY_LINES * 6);
      setActivity(buildActivityLines(records));
  }, ACTIVITY_POLL_MS);
  ```
  The wire client has only a `logs()` snapshot fetcher; no `/api/logs/stream` EventSource is opened. The feed DOES update without manual refresh (G-3 / the user-visible intent is met) and DOES render real, filtered, newest-first events — but via poll, not SSE.
- **Impact:** Functionally honest and self-refreshing, but a literal reading of c-AC-2 ("via `/api/logs/stream` (SSE)") is not met — the SSE transport the PRD names is not used. c-AC-1 (real filtered events, newest first) and c-AC-6 (live verification) are unaffected.
- **Recommendation:** Either subscribe to the existing `/api/logs/stream` SSE (the infra already exists per `mountLogsApi`, and the PRD-024 LiveLog panel consumes it) to satisfy the criterion literally, or amend c-AC-2 to accept a short-interval poll as the chosen follow mechanism. Low effort either way; flag for the plan author (`library-worker-bee`) if the poll is intentional.

## Suggestions (consider improving)

### S-1 — Substrate-only `shared` rows list by `honeycomb_id` when the substrate carries no `name` (cosmetic)

- **Location:** `src/daemon/runtime/dashboard/sync-api.ts:255-272` (`unionFor`, step 1).
- **Detail:** When a `shared` row has no live disk presence to supply a human name, the union falls back to `displayName = row.honeycombId` (an `hc_<hex>` token). This is honest and non-secret, but a teammate's shared-but-unpulled asset can list as `hc_abcd…` rather than a readable name until pulled. The substrate `name` column is read (`reduceCurrent` captures `r.name`) but is often empty for these rows. Consider surfacing the asset's frontmatter name into the substrate write so the union has a human label pre-pull. Not an AC violation (AC-1 requires the row to appear once with honest state — it does).

### S-2 — `resolveViewerAuthor` local-mode fallback returns `defaultScope.workspace` as the author token (sync-mount.ts:176-183)

- **Detail:** In local mode the viewer's author for `authoredByMe` is derived from the workspace name. This is a documented, presentation-only heuristic (it never partitions storage) and keeps dogfood Demote usable for the single local user, but it conflates "workspace" and "author identity." Acceptable for v1 local-mode; worth a note if multi-author local ever lands. No security impact — the daemon-side `authoredByCaller` gate (sync-api.ts:438-458) independently enforces author-only demote against the substrate `author` column regardless of this display flag.
