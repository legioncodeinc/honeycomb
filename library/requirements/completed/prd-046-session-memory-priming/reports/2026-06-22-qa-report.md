## Warnings (should fix)

- [ ] **Durable-key *sharpener* deferred — durable keys are content-derived, not purpose-distilled** — `src/daemon/runtime/summaries/prime-keys.ts:134-143`

  Episodic keys (`memory.key`) are generated by a dedicated grounded two-step gate pass and sharpness-checked (`key.ts`, `key.test.ts`). Durable keys (`memories.key`) have the column + a pure-SQL read, but when a legacy durable fact has no derived `key` the prime falls back to the raw `content` collapsed to one line — there is no dedicated durable-key distillation prompt. b-AC-2 ("every summary/fact has a ≤1-sentence keyworded key") is **functionally met** because `memories` rows are already distilled, concise facts whose `content` reads acceptably as a key, so every durable fact is primeable today. This is a quality-refinement gap, not a coverage gap. Parked as `046b-durable-key-sharpen` (low priority). Recommend running the grounded `redactSecrets`+key derivation over legacy durable rows in that follow-up.

  ```ts
  const fallback = typeof row.content === "string" ? row.content.replace(/\s+/g, " ").trim() : "";
  const key = rawKey !== "" ? rawKey : fallback;  // legacy durable fact → content-as-key
  ```

- [ ] **046f eval emits 2 of the 4 signals f-AC-2 enumerates** — `src/eval/prime.ts:20-43, 180-193`

  f-AC-2 lists four behavioral signals to emit: pull-through, redundant-search, **convergence**, **grounded-reference**. The harness emits only pull-through + redundant-search-reduction. The other two are honestly NOT scored (documented in `eval/README-prime.md`) because they require an LLM judge to be deterministic. This is an acceptable narrowing, not an AC violation, because (a) the parent index frames pull-through + redundant-search as the "at minimum" pair, and (b) f-AC-3 — the *gate* — requires beating cold "on at least the headline signal (pull-through and/or redundant-search reduction)," which is fully met. f-AC-2 is marked ⚠️ (partial-by-design) in the traceability table. Recommend adding the LLM-judge convergence/grounded-reference signals in a later slice if the headline pair proves insufficient.

- [ ] **046d Cursor binary upgraded from a `bootHarness` stub to the shared hook runtime — broader than "just the prime"** — `harnesses/cursor/src/index.ts:1-52`

  The Cursor harness binary was a legacy `bootHarness`-only stub that never drove the shared hook runtime, so the session-start prime literally could not be injected on Cursor without this change. The Bee upgraded it to `runHookBinary` (mirroring claude-code), which is the **minimum necessary** to satisfy d-AC-2. Side effect: capture + recall now ALSO flow on Cursor (strictly additive, CI-green). This is a necessary fix, not gratuitous scope creep, but it widens Cursor behavior beyond the prime. Recommend reconciling if a separate PRD claims ownership of the Cursor binary runtime wiring (already logged in the execution ledger).
