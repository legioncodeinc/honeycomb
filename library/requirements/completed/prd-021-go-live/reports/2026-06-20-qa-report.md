## Warnings (should fix)

- [ ] **`mountLogsApi` and `mountDashboardHost` are not called by the production `assembleDaemon()`**, `src/daemon/runtime/assemble.ts:54-57,297-320`

  The production composition root `assembleDaemon()` fires exactly four seams (verified at lines 297-320 of `assemble.ts`): `attachHooksHandlers`, `mountDashboardApi`, `mountNotificationsApi`, and `attachSessionsPrune`. `mountLogsApi` (`src/daemon/runtime/logs/api.ts`) and `mountDashboardHost` (`src/daemon/runtime/dashboard/host.ts`) are NOT imported and NOT called by `assembleDaemon()`. They are called only by the integration tests that mount them manually:
  - `tests/integration/golden-path-live.itest.ts:160`: `mountLogsApi(booted.assembled.daemon, { logger: booted.assembled.daemon.logger })`
  - `tests/integration/dashboard-logs-live.itest.ts:63-67`: `mountLogsApi(...)` + `mountDashboardHost(...)`

  d-AC-2 states: "given the daemon, when it is ASSEMBLED, then the `/api/logs` handler reads the request-logger ring buffer." d-AC-3 states: "given `honeycomb dashboard`, when it runs, then it opens a real viewable dashboard host." As written, neither AC is satisfied by the production composition — a real operator who boots the daemon via `honeycomb daemon start` (which calls `runAssembledDaemon` in `src/daemon/index.ts`, which calls `assembleDaemon()`) gets no `/api/logs` and no `/dashboard` endpoint; both return the 501 scaffold fallback.

  **Ruling on the assembly gap:** This is a Warning, not a Critical blocker, for the following reasons that reduce severity: (1) security-worker-bee's F-1 finding documented that `mountDashboardHost` must NOT be wired into `assembleDaemon()` until the `x-honeycomb-org` header-trust hardening and `protect:true` group placement are resolved — so the correct final state depends on a security prerequisite; (2) `mountLogsApi` is already on a `protect:true` group per server.ts and does not carry the same security gate, making its absence from `assembleDaemon()` a pure incompleteness issue with no security rationale; (3) the GATED live itest proves both seams work correctly against a real daemon; and (4) D-3 designates local single-user mode as the dogfood target, which narrows the immediate operator impact. However, the PRD's stated thesis — "burning down the deferred-assembly debt" and "the production assembly calls this once" — is not yet complete for these two seams. Both were explicitly anticipated by the ledger comment at `assemble.ts:297-320` (the `mountDashboardApi` note says 021d/e fill handlers), but neither materialised as a call in the final composition root.

  **Required remediation:** Add `mountLogsApi(daemon, { logger })` to `assembleDaemon()` alongside the other four seams (no security prerequisite — the `/api/logs` group is already `protect:true`). Add `mountDashboardHost(daemon, { storage })` to `assembleDaemon()` AFTER the security F-1 prerequisite is resolved: either gate it to `local` mode (`if (config.mode === "local") mountDashboardHost(...)`) so it is never reachable in team/hybrid, or place it on a `protect:true` route group. The wiring of `mountLogsApi` is safe to do immediately; `mountDashboardHost` should wait for the F-1 remediation wave.

  ```typescript
  // src/daemon/runtime/assemble.ts — in assembleSeams(), after the four existing calls:
  // (safe to add immediately, no security gate needed — /api/logs is already protect:true)
  mountLogsApi(daemon, { logger });
  // (add after security F-1 remediation, with local-mode gate or protect:true group)
  if (config.mode === "local") mountDashboardHost(daemon, { storage });
  ```

## Key Adjudications

### The assembly gap (d-AC-2, d-AC-3): Warning, not Critical

The question posed in the audit brief was: "Is this a BLOCKING AC gap — or acceptable given the local-mode/team-mode authz coupling security flagged?"

**Ruling: Warning (not blocking).** The two seams are proven to work live (the gated itest mounts them on a real assembled daemon and runs successfully). The gap is specifically in the production `assembleDaemon()` call sequence, not in the seam implementations. The blocking test for "Critical" is: would a real operator who runs `honeycomb daemon start` and then `honeycomb dashboard` today get a broken experience? Yes — they would get a 501 for `/dashboard` and no `/api/logs`. That is a real operator-visible incompleteness. However, the security prerequisite for `mountDashboardHost` (F-1) is a genuine coupling that the security-worker-bee correctly identified as requiring a separate remediation wave before it is safe to wire into the production assembly in team/hybrid mode. A "Critical blocks ship" ruling would be unwarranted given that:

1. The D-3 dogfood target is local single-user (not the team/hybrid path the security risk applies to).
2. `mountLogsApi` has no security gate and can and should be wired immediately.
3. The end-to-end proof of both seams is present via the gated itest.

The precise remediation is stated in the Warning above: wire `mountLogsApi` now (safe, no security gate), and wire `mountDashboardHost` after the F-1 remediation wave (gate it to `local` mode OR give it a `protect:true` group).

### The headline f-AC-3 cross-session recall: GENUINE LIVE PROOF

`tests/integration/golden-path-live.itest.ts:281-301` is a real live assertion, not a fake. The proof at lines 288-301 confirms:
- `RECALL_TERM` is a per-run unique nonsense token seeded ONLY into session ONE's captured turn.
- The recall query (`runCrossSessionRecall`, lines 387-424) is a real UNION ALL over `sessions.message` and `memory.summary` via production SQL guards (`sqlIdent`/`sqlLike`/`sLiteral`), run through the daemon's OWN live storage client.
- The assertion at line 293 checks `recall.hits.length > 0` and at line 296 checks that `recalledPaths` contains `pathOne` (session ONE's path, which session TWO never captured).
- The assertions at lines 300-301 confirm BOTH arms (`sessions` and `memory`) surfaced the term.
- recall-hit = 1.00 (2/2 arms), confirmed across 3 consecutive live runs.

This is the product thesis proven live.

### 021c workspace-partition regression: CONFIRMED

`tests/hooks/runtime/daemon-client.test.ts:175-208` asserts:
1. A credentials file with `workspace: "honeycomb_ci"` (a non-default workspace) is read back verbatim as `cred?.workspace === "honeycomb_ci"` (line 193).
2. The production `DaemonHookClient` stamps this workspace onto BOTH `x-honeycomb-workspace` header (line 205) AND `body.metadata.workspace` (line 207).

The regression is documented at lines 176-179: the original reader dropped the file's `workspace`, causing captured rows to land in the `default` partition while read-backs queried the real workspace — invisible on read-back. The fix is proven by the assertion at line 193 and the end-to-end wire-through at lines 197-207. Confirmed complete.

### 021e "imports but doesn't serve" finding: GENUINELY CLOSED

`tests/integration/mcp-transport-live.itest.ts` (ledger: 1/1 pass, no token needed) spawns the BUILT `mcp/bundle/server.js` as a stdio subprocess, drives a real JSON-RPC `initialize` + `tools/list`, and asserts `serverInfo.name === 'honeycomb'` plus tool capability. The itest file exists and the ledger confirms exit 0. The "imports clean but never connects" state from before 021e is closed.

### 021f BLOCKED items: HONESTLY PARKED

f-AC-6's two blocked sub-items are:
1. The recorded demo (requires a human operator with a real Claude Code GUI session and screen-recording capability).
2. The interactive Claude Code marketplace install / live editor session (requires a human to perform the real marketplace install).

Both are explicitly marked BLOCKED in the ledger at the f-AC-6 row with a "PRECISE HUMAN ASK" section (library/ledger/EXECUTION_LEDGER-prd-021.md:191-209). Per D-6 ("honest deferral allowed for the long tail... any part needing a truly interactive marketplace install or a screen recording that is NOT scriptable in this environment is parked BLOCKED with a specific ask"), this is the correct disposition.
