## Warnings (should fix)

- [ ] **`entity_attributes` provenance: `source` and `proposalId` fields are silently discarded on the row today**, `src/daemon/runtime/ontology/supersede.ts:275-277` and `src/daemon/runtime/ontology/entity-model.ts:326-328`

  The plan (a-AC-3, 008a FR-4) requires provenance carrying "memory id, source, and proposal id". The `memory_id` column is correctly written on every row. However `provenance.source` and `provenance.proposalId` are explicitly voided at the write sites (`void a.provenance.proposalId; void a.provenance.source;`) with a comment that these will ride the proposal row + deterministic id lineage and land when the columns are added by a future heal. The `entity_attributes` table has no `source`/`proposal_id` columns in the PRD-003b catalog today. The plan AC reads "provenance back to the memory and proposal", which is partially satisfied (the proposal row + id lineage are reconstructable) but not first-class on the attribute row itself. This is a documented forward-compatible choice, not an oversight, and the security auditor confirmed it. Severity is Warning (not Critical) because the plan's open question and the non-goals explicitly defer first-class column adds to additive schema heals, and the lineage is reconstructable. Recommend noting in the PR description that full `proposal_id` column lineage is deferred to a future additive heal.

  ```typescript
  // supersede.ts:275-277
  void a.provenance.proposalId;  // proposal link deferred â€” no column yet
  void a.provenance.source;
  ```

## Supersede Fix Soundness Assessment (b-AC-1/b-AC-2)

The in-place UPDATE `mark-prior-superseded` that failed live (active=2 expected 1) has been fully replaced. The fix is genuine, not papered over:

1. **Mechanism**: `appendPriorSuperseded` (`supersede.ts:304-351`) is a pure INSERT â€” same `id` as the prior, version bumped by 1, `status='superseded'`, all other columns copied forward unchanged. No UPDATE statement is constructed anywhere in the supersede path.
2. **Unit proof**: `supersede.test.ts:86-87` asserts `transport.requests.filter(r => r.sql.toUpperCase().startsWith("UPDATE")).toHaveLength(0)` â€” the fake transport captures every SQL; zero UPDATEs means the entire supersede path is append-only.
3. **Read convention**: `readCurrentStateById` (`supersede.ts:365-392`) and the live itest `scanRows` both resolve current state as highest-version-per-id, not raw status counts. This is the same convergent pattern `graph-persist.ts` uses.
4. **Live determinism**: `PRIOR_POLLS = 8` poll-convergent reads ensure a stale segment cannot cause a false-absent; polling only turns false-absent into true-present, never invents a row.
5. **Live confirmation**: Ledger records 3+ consecutive clean live runs (`supersede + deps + apply, 4 tests`) after the fix; live itests read back exactly 2 rows (1 active v2, 1 superseded v1) â€” the append pattern is durable on the real backend.
