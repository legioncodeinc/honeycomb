## Suggestions (consider improving)

**S-1 — b-AC-7: audit gate does not scan test files by design; the exemption list documents only `sql.ts`**

`scripts/audit-sql-safety.mjs:49`: `EXEMPT_BASENAMES = new Set(["sql.ts"])` exempts the helper module itself, but test files are skipped via the `.test.ts` suffix filter at line 334 (`!entry.name.endsWith(".test.ts")`). This is correct behavior — test files do not ship. However, the gate's header comment says "only production builders are gated" without naming tests. Consider adding a one-line comment at line 334 clarifying that `.test.ts` files are out of scope by intent, so a future reader does not think it is an oversight. This is cosmetic; the behavior is correct and b-AC-7 is fully met.

**S-2 — c-AC-2 test: introspection SQL passes `workspace` as empty string in the test fixture**

`tests/daemon/storage/heal.test.ts:102`: `buildIntrospectionSql` is called via `healColumns`, which passes `scope.workspace ?? ""` (`heal.ts:102`). The test's `SCOPE` object sets `workspace: "ws1"`, so the introspection SQL does filter by workspace — this is correct. However, the c-AC-2 test assertion (`altered.length === 1`) does not verify the introspection SQL carries the workspace filter. This is a test-depth observation, not a code defect; the production path is correct. Consider adding one assertion: `expect(fake.requests.find(r => /information_schema/.test(r.sql))?.sql).toMatch(/table_schema = 'ws1'/)` to pin the workspace-scoping property.

**S-3 — FR-3 (002a): `a-AC-5` non-daemon list does not include `src/shared`**

`tests/daemon/storage/invariant.test.ts:24`: `NON_DAEMON_ROOTS` does include `"src/shared"`. On inspection the list correctly covers `["src/cli", "src/daemon-client", "src/shared", "mcp", "harnesses", "embeddings"]`. No gap — this is a pre-flight observation that confirms correctness and requires no action.

## Five-Axis Evaluation

### Completeness

All 35 granular ACs and all 4 index roll-up ACs are implemented. Each AC maps to at least one named Vitest test whose describe/it label cites the AC ID. Seven test files, 69 tests, 7 passed. No AC is untested or vacuously tested.

The one structural gap noted: `b-AC-2` has two test-level entries in `sql.test.ts` (the `sqlIdent` test at line 31 and the `sqlColumnList` test at line 85), both labeled `b-AC-2`. The second test is a post-security-remediation addition (`sqlColumnList` was the F-1 fix). This is not a defect — both tests are substantive — but the label reuse is cosmetically awkward. Captured as Suggestion S-1 above.

### Correctness

Implementation code was read for all five modules (`sql.ts`, `config.ts`, `client.ts`, `heal.ts`, `writes.ts`, `vector.ts`, `schema.ts`). Each function's behavior matches the AC specification:

- `sqlStr` order-of-operations is correct (backslashes doubled before quotes, then quotes doubled, then control chars dropped) — the injection-safety order matters and was verified at `sql.ts:42-48`.
- `classifyFailure` checks permission patterns first before relation/column patterns, preventing the auth-masking misclassification described in c-AC-3 (`heal.ts:53-70`).
- `withHeal` retries exactly once by construction (`heal.ts:170-194`): the return of `runWrite()` on line 193 is terminal — there is no loop.
- `selectBeforeInsert` re-verifies without a `LIMIT` on the post-insert query (`writes.ts:270`), which is correct — a LIMIT 1 would hide a race.
- `buildVectorSearchSql` uses `ARRAY_LENGTH(emb, 1) > 0` to exclude null/empty-embedding rows from the vector branch, not a `IS NOT NULL` check — correct because an empty array `{}` also degrades (`vector.ts:201`).
- Score normalization formula `(1 + cosine) / 2` maps [-1,1] to [0,1] correctly. The `toScoredIds` function further clamps to `[0,1]` as a guard for out-of-range engine returns (`vector.ts:243`).

### Gaps

No functional gaps were found. The honest-gap scrutiny of live-flavored ACs:

- **e-AC-1 (GPU search)**: The implementation emits the `<#>` cosine operator against the tensor column and returns scored IDs via the fake transport. No live GPU hardware is exercised; the ledger documents this as the correct posture absent credentials. The contract (correct SQL shape, correct result mapping, dim validation before emit) is fully proven.
- **d-AC-3 (concurrent edits)**: Simulated by monotonically advancing `currentMax` state in the fake transport (`writes.test.ts:105-111`). This is accurate contract-level verification — the version-bump pattern's correctness does not depend on real concurrency; it depends on the INSERT-not-UPDATE guarantee, which is proven by code inspection and the no-UPDATE assertion.
- **c-AC-2 (information_schema)**: The fake transport returns canned column rows. The implementation issues the correct `SELECT column_name FROM information_schema.columns WHERE table_name = '...' AND table_schema = '...'` statement (`schema.ts:124-130`), and the heal engine diffs the result correctly. Contract-level proof is honest and appropriate.
