## L-1 Disposition (security hand-off)

**Issue (Low):** In `src/daemon/runtime/memories/store.ts` `mutateMemory`, the version-bump mutation (`applyControlledWrite`) lands BEFORE the `memory_history` audit row is written. If the audit INSERT fails after the mutation, the mutation is already durable (append-only DeepLake — no rollback).

**Decision: acceptable, documented Low — small/safe hardening applied (no behavior change).**
- The `audited` flag is SURFACED end-to-end (`api.ts:255,273` → response body), so a mutation whose provenance row failed is **observable, never silent**. The accountability requirement — the reason-gate — fires at the zod boundary AND in `store.ts` BEFORE the mutation, so a reasonless mutation can never land regardless of audit outcome. Audit-first ordering would be a worse asymmetry (an audit row for a mutation that then fails) on an append-only backend.
- **Remediation in-session (small + safe):**
  - `src/daemon/runtime/memories/store.ts` — added an ordering-rationale comment at the audit call site documenting the L-1 posture (mutate-first, audit-failure-surfaced-not-swallowed).
  - `tests/daemon/runtime/memories/store.test.ts` — added regression test `"a-AC-4 / L-1: a failing memory_history audit write after the mutation surfaces audited:false (not silent)"`: a responder that throws on the `memory_history` INSERT but succeeds on the `memories` version-bump proves the mutation lands (`action: version_bumped`, `memories` INSERT present) AND `audited === false` is surfaced (not swallowed). Passes (store.test.ts now 7 tests).
