## Suggestions (consider improving)

### S-1 — `resolveDeeplakeToken` is a tested, exported, but production-dead resolver

- **Where:** `src/daemon/runtime/vault/migrate.ts:113` (definition), exported via
  `src/daemon/runtime/vault/index.ts:70`, documented in `CONVENTIONS.md:69`. Consumed ONLY by
  `tests/daemon/runtime/vault/vault.test.ts:242-275`. A repo-wide grep for `resolveDeeplakeToken`
  finds NO caller in `src/` outside its own module + the barrel.
- **What:** The boot migration (`assemble.ts:1083-1097`) COPIES `~/.deeplake/credentials.json` →
  vault `secret/DEEPLAKE_TOKEN` at every boot. But nothing in production ever reads that copy
  back: the live DeepLake connection resolves its token through `loadCredentials` /
  `loadDiskCredentials` (`auth/credentials-store.ts`), which is **env → file only** — it has no
  vault arm. So the vault `DEEPLAKE_TOKEN` record is write-only in production; the
  `vault → env → file` precedence that `resolveDeeplakeToken` implements never runs outside tests.
- **Is this an AC-3 gap? No.** AC-3's literal text (index lines 133-139; a-AC-4/a-AC-5) requires:
  (1) the token is COPIED into the vault — **done + proven** (vault.test.ts:202-230, assemble.test.ts:1238);
  (2) the plaintext file is byte-unchanged + still loadable — **done + proven**;
  (3) "a login resolves from the vault" and "vault empty still resolves from file" — **proven at the
  resolver unit level** (vault.test.ts:242-269), which is what a-AC-5 asks for ("Proven by a test
  that … a login resolves from the vault"). The PRD does **not** require the live storage
  connection itself to resolve through the vault — and D-3 explicitly says the opposite:
  *"the plaintext file remains … authoritative for the SHARED Hivemind login until a later PRD
  changes that contract"* (index lines 90-93). The ledger documents this deliberately
  (`assemble.ts:1071-1082` comment: "the vault is an ADDITIVE cache"). **So the implementation
  matches the plan.** I confirmed the design choice rather than treating it as a defect, per the
  task's explicit instruction to cite the PRD either way.
- **Why still flag it:** an exported function with no production consumer is a latent honesty smell.
  Today it is justified (it is the seam a future PRD wires when it makes the vault authoritative).
  Recommend either (a) a one-line CONVENTIONS/AC-3 note that `resolveDeeplakeToken` is intentionally
  staged-not-wired pending the follow-up PRD, or (b) wiring `loadDiskCredentials` to consult the
  vault arm if the team wants AC-3's "vault wins" precedence to be true in the live login path now.
  No code change is required to ship PRD-032.

### S-2 — vault provider override always uses the Anthropic transport

- **Where:** `src/daemon/runtime/inference/model-client-factory.ts:170` —
  `applyProviderModelOverride` rewrites `account.provider` to the vault provider (e.g. `openai`),
  but `buildInferenceModelClient` then unconditionally builds `createAnthropicTransport()`.
- **What:** A vault `activeProvider=openai` (or `openrouter`) selection rewrites the routing
  config's provider/model strings, but the actual wire transport is still Anthropic. This is a
  **pre-existing PRD-010 limitation** (only the Anthropic transport ships), not introduced by
  PRD-032, and AC-6 only requires that the vault selection WINS over `agent.yaml` for the
  provider/model pair — which it does. The catalog gladly offers `openai`/`openrouter`, so a user
  can select a provider whose transport isn't yet implemented. Flagging for visibility; out of
  PRD-032's scope to fix (the catalog is D-6's curated list; transport breadth is PRD-010's).
