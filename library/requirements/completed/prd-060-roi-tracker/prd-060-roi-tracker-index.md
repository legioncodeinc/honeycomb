# PRD-060: ROI Tracker

> **Status:** Backlog, draft (2026-06-26). 6 sub-PRDs, not started.
> **Priority:** P1 (the credibility surface, proves the memory layer pays for itself; gates retention and the upgrade pitch)
> **Effort:** XL (> 3d, spans a foundational capture-pipeline build, two cost engines, an outbound billing client, a shared cross-device spend ledger + teams roster, and a new dashboard page)
> **Schema changes:** Additive only. New token/cache columns on the existing `sessions` catalog group via additive schema healing (060a). A new provider→model rate table lives in source, not the DeepLake catalog. **Two new tenant-scoped DeepLake tables, `roi_metrics` (append-only spend ledger) + `teams` (roster), via additive schema healing (060f).**

---

## Overview

Honeycomb captures every coding session and injects memory into every turn, but it has **no way to show whether that machinery pays for itself**. The operator (and every prospective upgrader) has to take the value on faith. This module builds an **ROI Tracker** page on the daemon-served dashboard that turns the faith claim into a ledger:

```
Net ROI = estimated LLM $ saved - (DeepLake infra cost + Honeycomb pollination cost)
  estimated LLM $ saved = cache savings (MEASURED) + memory-injection savings (MODELED estimate)
  DeepLake infra cost   = DeepLake billing API (compute GPU-hours / storage / transfer, in cents)
  pollination cost      = DeepLake embedding+ingestion+query GPU sessions (billing API itemizes by session_type)
                          + Honeycomb's own Haiku skillify token cost
```

The pivotal framing, and the thing that makes this hard, is that the ledger spans **two cost worlds that never see each other**:

> **World 1, the LLM tokens the user's coding assistant burns.** Claude Code / Codex / Cursor pay Anthropic/OpenAI per token. This is where Honeycomb *creates value*: prompt caching returns cache-read tokens at 0.1× input rate, and memory injection (modeled) lets the assistant reach an answer in fewer turns. **DeepLake never sees these tokens.**
> **World 2, the DeepLake GPU sessions Honeycomb's own machinery burns.** Every embedding, ingestion, and recall query is a DeepLake compute session billed in integer cents, itemized by `session_type`. Plus Honeycomb's *own* Haiku inference for the skillify loop ([PRD-016](../../completed/prd-016-skillify/prd-016-skillify-index.md)) is a token cost the daemon pays. **This is the cost side.** DeepLake's `/billing/*` API ([`api.deeplake.ai`](https://api.deeplake.ai/docs/doc.json)) exposes ~17 endpoints, all **infra** cost in integer cents, with **no token or cache notion**, it cannot tell you what the user's LLM cost was, only what DeepLake's GPUs cost.

The credibility crux runs through the savings half. **Cache savings are MEASURABLE**: `cache_read_input_tokens × (base_input_rate - cache_read_rate)` is arithmetic over billed fact, *if* we capture the per-turn token counts. **"Honeycomb saved you X overall" is a COUNTERFACTUAL**, a model of what the user *would* have spent without memory injection. The two must be presented as distinct kinds of number, and the modeled figure must **never** masquerade as a billed one. That honesty rule is the spine of 060b (the math) and 060e (the UX), and is itself an acceptance criterion below.

There is one foundational obstacle: **Honeycomb does not capture per-turn token data today.** The capture contract ([`event-contract.ts`](../../../../src/daemon/runtime/capture/event-contract.ts)) normalizes an assistant turn to `{ kind: "assistant_message", text }` and **discards the API `usage` object**; the Anthropic transport for Honeycomb's own inference ([`transport-anthropic.ts`](../../../../src/daemon/runtime/inference/transport-anthropic.ts)) parses only `content` and likewise drops `usage`; and no token/cost columns exist in any catalog table ([`src/daemon/storage/catalog/`](../../../../src/daemon/storage/catalog/sessions-summaries.ts)). The data **exists at the source**, Claude Code writes per-message `usage` (`input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`) to its transcript JSONL, so extracting it is a **build, not a research** problem. That build is sub-PRD 060a, and it gates the entire measured-savings half.

Two product decisions are **already locked** by the operator:

> **Locked 1, show BOTH.** The page shows measured cache savings (the green, defensible **headline**) *and* the modeled memory-injection savings (a clearly-labeled **estimate** with its assumption surfaced). Not one or the other.
> **Locked 2, capture is Claude Code first.** The token+cache capture pipeline (060a) is foundational and scoped to **Claude Code** (richest cache data) for v1; Codex/Cursor are explicit follow-ups. The ROI page must **degrade gracefully** when token data is absent, partial, or Claude-Code-only.

One further decision is locked: the spend record is **shared, not device-local.** The per-session ROI figure is pushed to a **shared DeepLake table** (`roi_metrics`) so ROI **aggregates across devices** and rolls up per-org, per-workspace, per-project, per-agent, and **per-team** (a real dimension, backed by a new `teams` roster). **Per-user is gated** on a verified backend user-claim that does not exist yet, Honeycomb has **no person identity today** (the DeepLake token is org-bound; `author` = `agent_id` = the machine), so a `user_id` column ships but stays empty until a verified `backend-token` claim populates it, with **no** self-asserted (git-email / OS-user) fallback. The local `/roi` dashboard reads this shared ledger at org/workspace scope governed by the existing `read_policy`. A **separate, hosted, multi-tenant admin surface** (cross-org reporting, per-user/team/org leaderboards) is **[PRD-061](../prd-061-hosted-roi-admin-surface/prd-061-hosted-roi-admin-surface-index.md)**, which deliberately flips this module's local-only posture and is its own surface.

The six sub-PRDs cover: the token/cache capture pipeline (060a, foundational), the cost+savings calculation engine and rate table (060b), the DeepLake billing API integration and infra read-model (060c), pollination cost metering (060d), the ROI Tracker dashboard page (060e), and the shared cross-device spend ledger + teams roster (060f).

## Goals

- A new **`/roi` dashboard page** on the existing loopback daemon-served dashboard that presents the Net-ROI ledger above, reading from a daemon read-model, one registry entry + one page component per [adding-a-page.md](../../../knowledge/private/dashboard/adding-a-page.md), touching **no** sidebar or router file by hand.
- **Measured cache savings as the headline**, computed from real captured `cache_read_input_tokens` against a maintained provider rate table, a defensible, billed-fact number.
- **Modeled memory-injection savings as a clearly-labeled, separate estimate**, carrying its tunable assumption as a **data field** (not buried in code) so the page can disclose exactly what the estimate rests on.
- **DeepLake infra cost** (compute / storage / transfer) read fail-soft from the DeepLake `/billing/*` API by a daemon-side, creds-gated client, cached behind a TTL, with the `session_type` breakdown (`query` / `embedding` / `ingestion`) feeding pollination cost.
- **Pollination cost** = Honeycomb's own Haiku skillify token cost (metered at the inference transport) + the DeepLake embedding/ingestion/query GPU-session cost (from the billing read).
- A **per-section status model** so the page never lies: every section reports `ok` / `partial` / `absent` / `unreachable` / `unauthenticated`, so a *measured $0* is visibly different from *unknown*, and the net is **never computed from incomplete inputs**.
- **Graceful degradation by construction:** Claude-Code-only token data shows a "Claude Code only" partial badge; an unreachable billing API shows a dash glyph for the affected line and a scoped retry, never a fabricated number; first-run shows dash-glyph placeholders, not a misleading `$0.00`.
- **Daemon is the sole egress.** Billing credentials live only in the daemon; the page holds no credentials and makes no outbound call, it reads a composite daemon read-model over loopback.

## Non-Goals

- **A hosted or multi-tenant ROI *surface*.** The spend *ledger* is now shared and cross-device (060f), but the **PRD-060 dashboard remains the loopback, local-mode-only surface** (mounted only when `mode === "local"`, parity with [`host.ts`](../../../../src/daemon/runtime/dashboard/host.ts)); `/roi` adds no public bind and no remote endpoint, it reads the shared ledger only at its own org/workspace scope under `read_policy`. The **authenticated, cross-org, partition-crossing admin surface** (per-user/team/org leaderboards) is a deliberate posture flip and lives in **[PRD-061](../prd-061-hosted-roi-admin-surface/prd-061-hosted-roi-admin-surface-index.md)**, not here.
- **Per-user attribution before a verified backend claim.** `roi_metrics` ships a `user_id` column (060f), but it is populated **only** from a verified `backend-token` claim and stays empty until that backend dependency lands; **no** self-asserted (git-email / `$USER` / OS-login) fallback, and **no** historical backfill. Per-user rollups are unavailable until then; per-**team** (a real `team_id` dimension + `teams` roster) is **not** gated on this.
- **Reinventing billing or becoming a billing system.** We *read* DeepLake's billing API and *compute* a derived ledger; we do not invoice, meter for charge, or reconcile against DeepLake's own statements.
- **A provider-pricing oracle.** Savings math rides a **maintained rate table** (a handful of providers/models, cents per Mtok), not a live pricing feed. Stale rates are a maintenance task, not a runtime dependency, and the table's source is surfaced so a wrong rate is auditable.
- **A new capture harness or a re-architecture of capture.** 060a *extends* the existing capture contract to carry the `usage` object it currently discards; it does not introduce a new intake path.
- **Codex / Cursor token capture in v1.** Explicit follow-ups (060a names them); v1 is Claude Code only, and the page degrades for the rest.
- **A charting dependency.** The trend chart is an inline SVG in the house GraphCanvas idiom ([`panels.tsx`](../../../../src/dashboard/web/panels.tsx)), no new npm chart lib.

## Data model changes

**Additive only. Two new DeepLake tables (060f); no destructive migration. Every new NOT NULL column carries a DEFAULT (`validateColumnDefs`).**

- **New tenant-scoped `roi_metrics` table** (060f), the shared, append-only, cross-device spend ledger, co-located with the telemetry-class tenant tables (`telemetry_counters`/`recall_qa_ledger`) in [`tenancy.ts`](../../../../src/daemon/storage/catalog/tenancy.ts) (`scope: "tenant"`). One immutable row per session via [`appendOnlyInsert`](../../../../src/daemon/storage/writes.ts); explicit `org_id`/`workspace_id`/`agent_id`/`project_id`/`team_id`/`user_id` identity columns; measured usage tokens (BIGINT); separate `measured_cache_savings_cents`, `modeled_savings_cents` (+ `modeled_assumption_ref`), `gross_cost_cents`, `infra_cost_cents` (all **BIGINT cents**); `cost_basis` (`'measured'｜'allocated'｜'none'`) + `allocation_method`; `price_ref` provenance; `period_start`/`period_end`/`created_at`. **`user_id` is GATED**, `''` until a verified `backend-token` claim, no fallback, no backfill. Re-price = append a new row with a new `price_ref`, resolve by `MAX(created_at)` per `session_id`; **never UPDATE.** Lookup indexes on `org_id`/`workspace_id`/`team_id`/`period_start` (+ drill-down `project_id`/`user_id`); **no embedding column, no JSONB, no BM25, no vector.**
- **New tenant-scoped `teams` roster table** (060f), `scope: "tenant"`, **version-bumped** writes ([`appendVersionBumped`](../../../../src/daemon/storage/writes.ts)); one row per (team, member) with a `member_type` `'agent'｜'user'` union (agent rows work today, user rows inert until `user_id` verified), plus `team_name`/`member_id`/`role`/`active`/`org_id`/`workspace_id`/`version`/`created_at`/`updated_at`. `team_id` on `roi_metrics` is resolved by a lookup against this roster at ROI-write time.
- **New token/cache columns on the `sessions` catalog group** ([`sessions-summaries.ts`](../../../../src/daemon/storage/catalog/sessions-summaries.ts)) added via **additive schema healing** (060a): per-turn `input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens`, plus a `source_tool` discriminant so Claude-Code rows are distinguishable from (future) Codex/Cursor rows. They follow the existing additive-heal posture, a missing column on a legacy dataset degrades the read to "token data absent", it never throws (AC-9). The single-source schema lives in [`src/daemon/storage/catalog/`](../../../../src/daemon/storage/catalog/index.ts); column adds go through the heal path, not a hand-written migration.
- **The provider→model rate table lives in source, not the catalog.** A maintained table (provider, model, `input_cents_per_mtok`, `output_cents_per_mtok`, `cache_read_cents_per_mtok`, `cache_write_cents_per_mtok`) authored in 060b. Whether it is a fresh module or reuses an existing rate/catalog location (e.g. the vault's [`catalog.ts`](../../../../src/daemon/runtime/vault/catalog.ts)) is an open question below. It is config-shaped data, versioned with the code, **not** a DeepLake row.
- **No persisted billing cache table.** The DeepLake infra read-model (060c) is an **in-memory TTL cache** in the daemon, not a new dataset, it re-reads the upstream `/billing/*` API on TTL expiry. The ledger is *derived*, never a stored ledger of record.

## API changes

The partition boundary and the auth backend are unchanged. New surface is split between **daemon loopback read-model routes** (consumed by the page) and **one outbound billing client** (consumed by the daemon). **No new public surface.**

- **Daemon (new loopback + local-mode-only read routes under the dashboard host group):**
  - `GET /api/diagnostics/roi`, a single **composite** read returning a `RoiView` with **explicit per-section status discriminants** (`ok` / `partial` / `absent` / `unreachable` / `unauthenticated`) so a measured `$0` is distinguishable from `unknown`. All money is integer cents; modeled savings carries its assumption as a field; `blendedCentsPerMtok` is `null` until capture is live.
  - `GET /api/diagnostics/roi/trend`, the time-series backing the inline-SVG trend chart (a separate, slower read).
  - These sit beside the existing diagnostics fetchers in [`api.ts`](../../../../src/daemon/runtime/dashboard/api.ts) and obey the same loopback + local-mode gate as [`mountDashboardHost`](../../../../src/daemon/runtime/dashboard/host.ts).
- **Daemon (new outbound billing client):** a creds-gated, fail-soft client for the DeepLake `/billing/*` endpoints ([`api.deeplake.ai`](https://api.deeplake.ai/docs/doc.json)), `GET /billing/summary`, `GET /billing/usage/compute` (the `session_type` breakdown), `GET /billing/account` (unit rates), and friends, new module [`src/daemon/runtime/dashboard/roi-billing.ts`](../../../../src/daemon/runtime/dashboard/roi-billing.ts), mirroring the injectable-`fetch` + retry-on-429/5xx + token-redaction posture of [`deeplake-issuer.ts`](../../../../src/daemon/runtime/auth/deeplake-issuer.ts). This is the **only** outbound call this module adds beyond the existing `api.deeplake.ai` auth traffic, and it is the **sole** holder of billing creds.
- **Page (new wire + contract shapes, no new egress):** wire schemas in [`wire.ts`](../../../../src/dashboard/web/wire.ts), contract types in [`contracts.ts`](../../../../src/dashboard/contracts.ts), a `roi.tsx` page in [`src/dashboard/web/pages/`](../../../../src/dashboard/web/pages/), and a registry entry in [`registry.tsx`](../../../../src/dashboard/web/registry.tsx). Two `usePoll` loops (billing slow ~60s, token per-session) per the frontend brief. The page is a **pure function of the view-model** and holds no creds.

## Open questions

- [ ] **Modeled-formula authority and tunability.** What *is* the memory-injection savings model (turns-saved × avg-turn-cost? a fixed % of measured spend?), where does its assumption live so it is one editable place, and who signs off on the exact assumption string the UX surfaces in the ⓘ popover + page-foot footnote? Until this is pinned, the modeled number is a placeholder. (Owned by 060b; UX disclosure in 060e.)
- [ ] **Measured/modeled disclosure sign-off.** The operator must sign off on (a) the exact assumption wording, (b) whether *any* cost input is itself an estimate (and must therefore also carry `est.`), and (c) the net-with-missing-input ruling, confirmed here as **do not compute net from incomplete inputs; show a dash glyph and a scoped retry**. This is the honesty contract; it is a gate, not a nicety.
- [ ] **Pricing-table source.** Fresh module vs reuse of an existing rate/catalog location (e.g. [`vault/catalog.ts`](../../../../src/daemon/runtime/vault/catalog.ts))? And what is the staleness/refresh policy, a dated constant with a "rates as of" stamp shown on the page? (060b.)
- [ ] **Trend backfill before capture-start.** The trend chart has no token history before 060a ships; does the trend show infra-only history with a "savings tracked from \<date\>" marker, or start empty? (060e + 060c.)
- [ ] **Billing API cadence, caching, and rate-limits.** TTL for the `/billing/*` read-model, behavior under 429, and whether the slow `usePoll` (~60s) is even needed given how slowly billing moves, possibly a much longer TTL with a manual refresh. (060c.)
- [ ] **Per-project vs workspace billing granularity.** Does `/billing/*` itemize per DeepLake project/dataset or only per workspace, and does that reconcile with Honeycomb's multi-project scope ([PRD-049](../../completed/prd-049-multi-project-and-context-switching/prd-049-multi-project-and-context-switching-index.md))? If billing is workspace-coarse, the page must say "workspace-wide" rather than imply per-project attribution. (060c.)
- [ ] **Negative-net framing.** A user who barely uses memory can show a negative net (infra+pollination > savings). How is that framed honestly without reading as "this tool costs you money", and `honey` brand color must **never** encode sign (positive = `var(--verified)`, negative = `var(--severity-critical)`). (060e.)
- [ ] **Harness rollout order.** Confirm Claude Code first (locked), then which of Codex / Cursor is the next capture target and what their transcripts expose for cache (Cursor in particular may not surface cache-read counts at all, capping measured savings to Claude Code). (060a.)
- [ ] **Cost-rising-must-not-render-green.** The prior-period delta on cost KPIs must invert the usual green=good sense (a *rising* cost is bad); confirm whether this needs an additive `Kpi` `sense`/`delta-label` prop, which would **escalate to design-system-worker-bee** if it exceeds an additive change. (060e.)
- [ ] **Verified backend user-claim, the per-user DEPENDENCY (gating).** Per-user rollups depend on a verified `backend-token` claim that **does not exist today**, Honeycomb has no person identity (the DeepLake token is org-bound; `author` = `agent_id` = the machine). What backend issues that claim, what is its shape, and how does the daemon obtain `claim.userId`? Until it lands, `roi_metrics.user_id` is `''` everywhere and per-user is inert by construction (no self-asserted fallback, no backfill). This is the single dependency that unblocks per-user here and the leaderboards in [PRD-061d](../prd-061-hosted-roi-admin-surface/prd-061d-hosted-roi-admin-surface-per-user-claim-gating.md). (060f; needs an auth/backend decision, surface to auth-worker-bee + security-worker-bee.)
- [ ] **Per-team roster authorship + lifecycle.** Who creates teams and assigns agents/users, and via what surface (a local dashboard admin page? the hosted [PRD-061](../prd-061-hosted-roi-admin-surface/prd-061-hosted-roi-admin-surface-index.md) surface? a CLI?)? Until a roster-authoring surface exists, `team_id` is `''` for everyone and the team rollup is empty even though the `teams` table is ready.
