## Suggestions (consider improving)

**S-1** `job-queue.ts:518, 571, 740` — The SQL builder wraps `this.tbl()` with additional double-quote delimiters directly in the template string (e.g. `` `FROM "${this.tbl()}"` ``). `sqlIdent` already validates the identifier; the surrounding `"..."` in the template is what produces the quoted SQL identifier. This is correct but relies on a caller convention that is not enforced by `sqlIdent` itself. A future refactor could have `sqlIdent` return the already-quoted form (e.g. `"memory_jobs"`) and remove the manual wrapping, eliminating the opportunity for a future caller to forget the quotes. Non-blocking; the current pattern is sound and the `audit:sql` gate already covers it.

**S-2** `src/daemon/runtime/services/runtime-path.ts` (claim-map) — Per the security report M-1, the claim map has no hard size cap. A pathological client could inflate it over many unique session IDs within the daemon's lifetime. The security report recommended a max-entries eviction policy. This is tracked but not a blocker for PRD-004.

**S-3** `src/daemon/runtime/services/job-queue.ts:DISCOVER_POLLS=8, RESOLVE_POLLS=8` — The convergence polling constants are hardcoded without documentation of the expected cost (number of round-trips per lease or reap call). A follow-up could expose them as `JobQueueConfig` knobs so a deployment with a reliably-fresh backend (e.g., a test double) can reduce the polling cost. Currently non-tunable without touching the constant.

## Notes on Queue Design Honesty

The version-bumped append-only redesign is a fair, documented architectural choice, not a hidden fragility. The rationale is stated directly in the file header (`job-queue.ts:25-48`) and in the catalog record (`runtime-jobs.ts:22-58`): independent live testing proved the backend serves scans and point-reads from segments of non-monotonically-varying freshness, making in-place UPDATE unreliable for status-filtered lease scans. The append-only pattern avoids the root cause — versions are append-only and monotone, so `MAX(version)` across a bounded union of polls converges up to the true current state. The convergence constants (`RESOLVE_POLLS=8`, `DISCOVER_POLLS=8`) are conservative; the orchestrator verified 5 consecutive live runs of 6/6 after the redesign. The one mild concern is S-3: the constants are not currently configurable, so a deployment with a reliably-fresh backend pays the same polling cost. This is a non-blocking optimization opportunity.
