## Suggestions (consider improving)

- [ ] **Embedding-deps install is implicit via `optionalDependencies`, not an explicit installer step** — `scripts/install/install.sh:118-133`, `scripts/install/install.ps1:103-122`, `package.json:127-128`

  a-AC-1 reads "installs the embedding deps." The implementation satisfies this correctly: `@huggingface/transformers` is an `optionalDependency` of `@legioncodeinc/honeycomb`, so `npm i -g` pulls it transitively (and a `--no-optional`/offline install degrades to the BM25 lexical fallback per PRD-025 D-4 — no quality cliff). This is the intended design and is documented in the script comments. The only refinement worth considering is a one-line progress-log step (`✓ embedding runtime available` / `→ embeddings will warm in the background`) so the user sees the embedding-deps step the AC names, rather than it being silent inside the npm install. Purely cosmetic; the AC is met as written.

- [ ] **`get.honeycomb.*` vanity domain + published checksum deferred (BLOCKED, non-gating)** — `scripts/install/install.sh:33-37`, `scripts/install/install.ps1:33-36`

  The `curl|sh` / `irm|iex` trust-posture follow-up (a vanity host + an "inspect before piping" checksum page) is intentionally deferred and tracked as a non-gating BLOCKED item per operator decision D1. The installer ships on the interim `https://raw.githubusercontent.com/legioncodeinc/honeycomb/main/scripts/install` URL, which fully satisfies a-AC-1/a-AC-5/a-AC-6. **Confirmed no AC secretly depends on the vanity domain** — every dashboard/daemon AC keys off the loopback `127.0.0.1:3850` contract, and `honeycomb.local` is best-effort-only (a-AC-6) with a tested loopback fallback. Recommend filing the vanity-domain + checksum work as a tracked follow-up before public launch (PRD-048 adjacency), but it does not block PRD-050 close-out.
