## 49e-AC-2 Judgment (the scrutinized item) ‚Äî PASS, with a documented partial-by-design boundary

**AC text:** *"Selecting a Project re-scopes the codebase graph, memory graph, memories, and sync pages to that `project_id` on the next render."*

**What the implementation does, per surface:**

1. **Memories + recall (the load-bearing half) ‚Äî TRUE row-level project filtering. PASS.** The Memories page threads `useScope().scope.project` into `wire.listMemories(limit, project)` and `wire.recall(query, project)` (`pages/memories.tsx:559-598,636-642`), which stamp `x-honeycomb-project` (`wire.ts:1275-1279,1243-1248`). The daemon resolves it via `resolveRequestProject` and ANDs the shared `buildProjectScopeClause` project predicate into the live list (`memories/api.ts:467-473`) and into **every** recall arm ‚Äî lexical, `<#>` cosine, hydrate, and native hybrid (`memories/recall.ts:210-274,595-676`; `hybrid-recall.ts:147-179`). A project-B memory is filtered server-side and cannot surface in project A even on a strong vector hit. This is the real isolation the AC's intent turns on, and it holds.

2. **Memory graph ‚Äî re-fetch-and-reframe, NOT row-level. Judged acceptable PASS.** The memory graph is derived from the `entities` and `entity_dependencies` **ontology** tables, which are **engine-scoped and carry no `project_id` column** (`catalog/knowledge-graph.ts:88-92,169` ‚Äî explicitly "NO `org_id`/`workspace_id` ‚Äî engine table"). The daemon memory-graph endpoint (`dashboard/api.ts:665-668` ‚Üí `fetchMemoryGraphView` ‚Üí `buildMemoryEntitiesSql:472-478`, `buildMemoryDependenciesSql:486-492`) has **no project predicate and does not read `x-honeycomb-project`**. The page DOES re-fetch on project change (`pages/graph.tsx:520-539`, effect keyed on `project`) and gates the needs-selection state, so the surface re-renders per the AC's "on the next render" ‚Äî but the underlying graph is workspace-level. **Judgment:** the question posed ("should the memory graph be project-filtered at the row level, and is it?") resolves to: it is **not**, and with the current data model it **cannot** be, because the ontology entities are not project-partitioned ‚Äî they are a workspace-level knowledge graph distinct from the project-stamped `memories` rows. Project-filtering the memory graph would require adding `project_id` to `entities`/`entity_dependencies` and threading it through the PRD-008 ontology pipeline ‚Äî a schema + pipeline change explicitly outside PRD-049's stated data model (the index scopes `project_id` to `memories`/`sessions`/skill rows). So this is a genuine data-model boundary, not an implementation miss. Acceptable.

3. **Codebase graph ‚Äî genuinely workspace-level by data model. PASS.** The codebase graph is read from a single LOCAL workspace snapshot (`/api/graph`, owned by `codebase/api.ts`), built per-workspace; it has no project dimension and the PRD never asserts one for it (the index keys "Repository-style" scope off `projectDir ?? process.cwd()`, not a per-project partition). Re-scoping here correctly manifests as re-fetch + reframe with the project header stamped (`wire.ts:1233-1235`). PASS.

4. **Sync ‚Äî view-level gate, no row filter (Suggestion).** The Sync page gates on project and re-hydrates on change, but `assetsView()` is not project-parameterized (workspace-level). Internally consistent with the graph posture; flagged as a Suggestion, not a gap.

**Verdict on 49e-AC-2:** **PASS.** The implementer's characterization is accurate and the right call. The load-bearing predicate (memories + recall) row-filters correctly; the two graph surfaces re-scope at the view level because their data is workspace-level by data model (codebase snapshot) or engine-level by schema (ontology entities). This satisfies the AC's intent ‚Äî "selecting a project re-scopes the project-specific surfaces on the next render" ‚Äî without over-claiming a row-level filter the ontology schema cannot support. **Not** a rubber-stamp: the divergence from the literal "to that `project_id`" wording is recorded as a Warning so it is visible, and the recommendation is explicit (if true memory-graph project-scoping is later required, add `project_id` to the ontology tables ‚Äî a follow-up PRD, not a 049 defect).
