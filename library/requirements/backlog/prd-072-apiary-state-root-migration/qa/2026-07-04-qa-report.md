## Critical Issues (must fix)

- [ ] **Skillify worker-lock paths never cut over: production still creates `~/.honeycomb/state/skillify/`**, `src/daemon/runtime/skillify/worker.ts:252-254` and `src/daemon/runtime/skillify/miner.ts:656-658`

  The index inventory (family 5) and the "Files expected to change" table both name `worker.ts:253` and `miner.ts:657` as skillify-state sites that move under the new root, but both `defaultLockBaseDir()` functions still hardcode the legacy dir. The daemon assembly builds the skillify worker without injecting a lock (`buildSkillifyWorker`, `src/daemon/runtime/assemble.ts:2021-2035`), so `worker.ts:285` falls back to `createFileWorkerLock(defaultLockBaseDir())` and every production skillify run recreates `~/.honeycomb/state/skillify/<projectKey>/worker.lock`. This breaches index AC-1 ("nothing honeycomb-owned is created under `~/.honeycomb/`" on a fresh install) and re-mints the legacy dir immediately after the skillify-state mover cleaned it. Remediation: resolve both defaults through `honeycombStateDir()` (matching `watermark.ts:38` and `config.ts:34`).

  ```ts
  // worker.ts:252-254 (miner.ts:656-658 is identical)
  function defaultLockBaseDir(): string {
  	return join(homedir(), ".honeycomb", "state", "skillify");
  }
  ```

- [ ] **Telemetry SQLite has no legacy-fallback read, and a failed mover permanently strands the legacy database**, `src/daemon/runtime/telemetry/fleet-store.ts:61-63` with `src/daemon/runtime/state-migration/families.ts:67`

  Index AC-3 and the 072b scope table item (3) require every family to read new-path-first, legacy-path-second while the window is open (US-072b.1.2: "Given any family's mover fails ... reads of that family come from the legacy path"). `fleetTelemetryDbPath` resolves only the new path, and `createDatabase` (`fleet-store.ts:222-229`) creates the file there when absent. Failure sequence: boot 1's mover fails (e.g. the `.sqlite` is locked by a lingering old daemon), the marker records `failed` (correctly retryable), but the store then mints a fresh empty database at the new path; boot 2's retry hits `moveFile`'s `if (existsSync(newFile)) return "skipped"` (`families.ts:67`) and the family is marked `complete`. The legacy database survives on disk (additive holds) but its rows are never migrated and never read again, and doctor's polling silently loses the install's telemetry history. The same skip-when-destination-exists trap applies to `moveDir` (`families.ts:99`). Remediation: either read-fall-back in `fleetTelemetryDbPath` (mirroring `preferExistingPath` as the machine key does at `secrets/store.ts:176-181`), or make the retry distinguish "destination minted fresh by a store" from "destination is the migrated copy" before marking complete.

  ```ts
  // fleet-store.ts:61-63: new path only, no legacy-second read
  export function fleetTelemetryDbPath(options: FleetRootOptions = {}): string {
  	return join(honeycombStateDir(options), FLEET_TELEMETRY_DIR_NAME, FLEET_TELEMETRY_DB_FILE_NAME);
  }
  ```

## Warnings (should fix)

- [ ] **Skillify state reads have no legacy fallback**, `src/daemon-client/skillify/config.ts:34-36`, `src/daemon/runtime/skillify/watermark.ts:38-40`, `src/daemon-client/skillify/manifest.ts:51-53`

  The 072b scope table grants every family "(3) a new-path-first legacy-second read while the window is open"; the skillify config, watermark, and manifest resolvers point only at the new path. If the `skillify-state` mover fails, config silently reverts to defaults and watermarks force re-mining even though the legacy files sit intact next door, and a store that lazily creates `state/skillify/` at the new path defeats the retry via `families.ts:99` exactly as in the Critical above. Graded Warning rather than Critical because 072b itself declares watermark loss "not data loss" and the content is regenerable, but the plan's read-fallback requirement is unmet for this family.

- [ ] **The legacy pid dual-stamp recreates `~/.honeycomb/` on a FRESH install, against AC-1's literal text**, `src/daemon/runtime/assemble.ts:787-796`

  `acquireSingleInstanceLock` dual-stamps the legacy pid file unconditionally whenever the legacy dir option is supplied (production always supplies it), including on a machine that never had `~/.honeycomb/`. Index AC-1 says a fresh install creates nothing honeycomb-owned under `~/.honeycomb/`; migration mechanics step 5.5's DEFAULT says dual-stamp during the window. The two plan statements conflict, and the implementation chose always-stamp (the security audit documented the same as Low, tied to window close-out). Recommended reconciliation: stamp the legacy pid only when the legacy dir already exists (upgrade installs keep operator/doctor continuity, fresh installs stay clean), and have library-worker-bee reconcile AC-1 with mechanics 5.5.

  ```ts
  // assemble.ts:791-793: unconditional mkdir of the legacy dir
  mkdirSync(options.legacyDir, { recursive: true, mode: DIR_MODE });
  writeFileSync(join(options.legacyDir, PID_FILE_NAME), pid, { encoding: "utf8" });
  ```

- [ ] **The migration trigger is daemon-assembly-only, narrower than the index mechanics**, `src/daemon/runtime/assemble.ts:2045-2053`

  Migration mechanics step 1 names the trigger as "first boot (daemon assembly or CLI verb that touches state)". The bootstrap runs only inside `assembleDaemon` (gated to the real assembly). A CLI verb that touches state before any daemon boot (e.g. `honeycomb asset register`) reads via fallback and writes to the new path, which then flips the asset-registry mover to `skipped`-complete without a real move. Content is preserved in that specific family (the store reads legacy-fallback and rewrites the full set), so impact is low, but the trigger contract is only partially implemented and the marker can record families complete that were never actually moved.

- [ ] **The index test plan's packaged-upgrade integration smoke was not shipped**, `library/requirements/backlog/prd-072-apiary-state-root-migration/prd-072-apiary-state-root-migration-index.md:156`

  The test plan calls for an integration smoke "mirroring `scripts/local-queue-packaged-upgrade-smoke.mjs`" (install old build, upgrade in place, confirm no double-start and correct state layout). No such script or test exists in the diff. The unit suites cover the pieces (lock continuity, movers, marker) but the end-to-end upgrade path is unproven. The "live proof" item on line 157 is a manual operator step and is out of code scope.

## Suggestions (consider improving)

- [ ] **Stale legacy-path documentation and a dead constant**, `src/daemon/runtime/secrets/contracts.ts:228`, `src/hooks/shared/session-start-seams.ts:43,102,197`, `src/daemon-client/skillify/manifest.ts:73`, `src/daemon/runtime/assemble.ts:1067`

  `MACHINE_KEY_DIR_NAME = ".honeycomb"` is no longer consumed by `secrets/store.ts` (which now resolves through `honeycombStateDir`) yet remains exported through `secrets/index.ts:23`. The session-start-seams doc comments still say the device id lives at `~/.honeycomb/device.json`, `manifest.ts:73` still says the base dir is "i.e. `~/.honeycomb`", and `assemble.ts:1067` still names `~/.honeycomb/nectar.json`. All are comment/dead-code drift, not behavior.

- [ ] **The notifications legacy read bypasses the injectable `StateFs` seam**, `src/notifications/state.ts:203-207`

  `preferExistingPath` probes with real `node:fs` `existsSync` while the store's own reads/writes go through the injected `StateFs`. In the one configuration where a caller injects `fs` but not `dir`, existence is answered by a different filesystem than the read. Harmless today (tests always inject `dir`), but routing the existence probe through the seam keeps the module honest.
