# PRD-051a: Memory Graph → Markdown Vault — the export engine

> **Parent:** [PRD-051](./prd-051-memory-graph-obsidian-vault-index.md)
> **Status:** Draft
> **Priority:** P2
> **Effort:** M
> **Schema changes:** None (reads existing ontology tables; writes LOCAL files)

---

## Overview

The core of PRD-051: a daemon-side writer that reads the PRD-008 ontology and emits an
**Obsidian-compatible Markdown vault** — one note per `entity`, `[[wikilinks]]` for `entity_dependencies`
relations, YAML frontmatter for the structured fields — into a stable per-scope directory. Stock
Obsidian opens the result with **no plugin**: the graph view, backlinks, and search all work off plain
Markdown + wikilinks. Reachable both from a CLI verb and a daemon endpoint (the latter is what 051b's
"Open in Obsidian" button calls).

The writer is the SINGLE owner of the on-disk vault format. It maps the same ontology data
`fetchMemoryGraphView` already serves (`src/daemon/runtime/dashboard/api.ts`) — so the vault and the
in-app graph never disagree on what the graph *is* — but renders it as a knowledge base instead of an
SVG.

## Goals

- A pure, tested mapping `ontology → VaultFiles` (entities + dependencies + mentions → notes + links),
  separable from the filesystem write so it is unit-assertable (mirrors `snapshotToGraphView`'s purity).
- Write a valid vault to the canonical per-scope dir, **idempotently**: re-export reflects the current
  graph and reconciles deletions (no orphaned notes for entities that no longer exist).
- Honest empty state: a `built:false` graph exports **nothing** and reports the "run pollinate" hint.
- XSS-/path-traversal-safe for untrusted entity text; no secret/token ever written.
- Expose it as `honeycomb vault export` (CLI) and `POST /api/vault/export` (daemon), returning the vault
  path so 051b can build the deep-link.

## Non-Goals

- The `obsidian://` deep-link / dashboard button — that is **051b** (this PRD just returns the path).
- Plugin-specific frontmatter, Dataview fields, or `.canvas` output — **051c** (optional/future). v1
  frontmatter is plugin-agnostic (plain keys any plugin *can* read, none *requires*).
- Ingesting user edits back into the ontology (the vault is a derived, overwrite-on-export projection).
- Bounding/capping node count. Obsidian handles large vaults; unlike the SVG canvas there is no render
  budget here (call this out explicitly — it is the inverse of the graph-cap decision).

## User stories

- *As an operator*, I run `honeycomb vault export` and get a folder of notes I can open in Obsidian and
  explore with the graph view + backlinks.
- *As a user*, when my memory graph is empty, the command tells me to populate it first rather than
  writing an empty/fake vault.
- *As a security reviewer*, I confirm a hostile entity name (`../../etc/passwd`, `[[`, `---`) cannot
  escape the vault dir, break frontmatter, or forge a wikilink.

## Vault layout (the format this PRD owns)

```text
~/.honeycomb/obsidian/<scope-key>/            ← canonical per-scope vault root (see OQ-2)
  Home.md                                     ← index/landing note (the 051b deep-link target)
  Entities/
    <Sanitized Entity Name>.md                ← one note per entity (id-suffixed on collision)
  README.md                                   ← "this vault is generated by Honeycomb; edits are overwritten"
```

Per-entity note shape (plain Markdown + YAML frontmatter, no plugin required):

```markdown
---
honeycomb_id: "ent_8f3a…"          # stable entity id (provenance, not shown in graph)
type: "project"                     # entity.type
source: "memories"                  # provenance source_type
agent: "claude-code"                # agent_id scope
updated: "2026-06-26T14:02:11Z"     # entity.updated_at
aliases: ["Honeycomb"]              # entity.name (so wikilinks by display name resolve)
tags: ["honeycomb/entity/project"]  # derived, plugin-agnostic
---

# Honeycomb

## Depends on
- [[DeepLake]] — *reason:* "stores all memory rows" (strength 0.9, confidence 0.8)

## Supersedes
- [[Old Storage Plan]]

## Related
- [[PRD-008]] — *reason:* "defines the ontology"

## Mentioned in
- `memories/prd-008-ontology` · `sessions/a7f3c`
```

- **Nodes → notes:** one note per `entity`; title = `entity.name`; frontmatter carries id/type/source/
  agent/updated (the fields `fetchMemoryGraphView` and the ontology tables expose,
  `src/daemon/storage/catalog/knowledge-graph.ts`).
- **Edges → wikilinks:** each `entity_dependencies` row becomes a `[[Target]]` under an H2 named for the
  relation `type` (`depends_on`→"Depends on", `supersedes`→"Supersedes", `related_to`→"Related",
  `mentions`→"Mentions"), annotated with the edge's `reason`/`strength`/`confidence`. Outgoing edges
  render on the source note; Obsidian derives the incoming side as automatic backlinks (no duplication).
- **Mentions:** `memory_entity_mentions` render as a code-spanned list of memory keys (not wikilinks —
  memories are not notes in this vault; OQ-3 revisits whether to emit memory notes too).

## Acceptance criteria

| ID | Criterion |
|---|---|
| a-AC-1 | The pure mapper turns a fixture ontology (N entities, M dependencies) into N entity notes + a Home note, with every dependency rendered as a `[[wikilink]]` under its relation heading. Unit-asserted with no filesystem. |
| a-AC-2 | Writing the vault produces a directory openable in stock Obsidian: the graph view shows the entities connected by the wikilinks; no plugin is needed for graph/backlinks/search. |
| a-AC-3 | A `built:false` (empty) graph writes NO vault and returns the honest "no memory graph yet — run `honeycomb pollinate trigger --compact`" message (parity with the in-app empty state). |
| a-AC-4 | Entity names are sanitized to safe single-segment filenames (no `/ \ .. : *` traversal/illegal chars); a name collision after sanitization is disambiguated by an id suffix; the original name is preserved in `aliases:` so wikilinks still resolve. |
| a-AC-5 | Untrusted entity text cannot break the file: frontmatter values are YAML-escaped/quoted, wikilink targets with `]]`/`[[`/`\|` are escaped or aliased, and a scan of the output contains no secret/token (reuse the dashboard no-secret guard). |
| a-AC-6 | Re-exporting reconciles: notes for entities no longer in the graph are removed (or moved to an `_archive/`), so a stale note never lingers. The op is idempotent for an unchanged graph (no spurious diffs). |
| a-AC-7 | `honeycomb vault export` (CLI) and `POST /api/vault/export` (daemon) both run the writer and report the vault root path + counts (`{ vaultPath, entityCount, edgeCount }`); the daemon writes daemon-side only (local FS), never the client. |

## Implementation notes

- **Read path:** reuse the ontology read the dashboard already uses — `fetchMemoryGraphView`
  (`src/daemon/runtime/dashboard/api.ts`) for the node/edge set, optionally enriched via the ontology
  read surface (`src/daemon/runtime/ontology/api.ts`, `/api/ontology/*`) for `reason`/`strength`/
  `confidence`/mentions that the slim GraphView drops. Decide one source of truth (OQ-4).
- **Pure mapper:** add `ontologyToVault(graph, opts): VaultFile[]` (a `{ path, contents }[]`) next to the
  writer, mirroring how `snapshotToGraphView` is pure + unit-tested. The filesystem writer is a thin,
  separately-tested shell (`writeVaultAtomic`), echoing `writeSnapshotAtomic`
  (`src/daemon/runtime/codebase/snapshot.ts`).
- **Canonical dir:** `~/.honeycomb/obsidian/<scope-key>/`, sanitized exactly like `defaultGraphBaseDir`
  (`src/daemon/runtime/codebase/api.ts:106`) so the slug can never traverse. Scope resolved via
  `resolveScopeOrLocalDefault` (`src/daemon/runtime/scope.ts`), fail-closed 400 outside local (mirror
  `mountGraphApi`).
- **Endpoint:** `mountVaultApi(daemon, …)` mirroring `mountGraphApi`; `POST /api/vault/export` →
  `{ vaultPath, entityCount, edgeCount, builtFalse? }`. On the RBAC-protected group.
- **CLI:** a new `src/cli/vault.ts` (`honeycomb vault export [--source memory] [--out <dir>]`), wired like
  the existing verbs (no graph CLI exists today — this is the first vault/graph CLI surface).
- **Safety:** filename sanitizer (allowlist `[A-Za-z0-9 ._-]`, collapse, trim, id-suffix on collision);
  YAML value quoting; wikilink target escaping; reuse the dashboard's secret-scan over the rendered
  output before write.
- **Reconciliation:** write to a temp dir then atomically swap, OR diff-and-prune the `Entities/` dir
  against the current entity-id set; never partial-write a half vault (crash-safe like the snapshot
  writer). Preserve `.obsidian/` (user's per-vault settings) across re-exports — never clobber it.

## Open questions

- [ ] **OQ-1 (source):** memory graph only for v1, or generalize `--source memory|codebase` now? (Parent
  module OQ.) Lean: memory-only v1, writer built source-agnostic internally.
- [ ] **OQ-2 (dir):** is `~/.honeycomb/obsidian/<scope>/` right, or should the user be able to target an
  existing vault as a *subfolder* (e.g. `<their-vault>/Honeycomb/`) so it coexists with their notes?
- [ ] **OQ-3 (memory notes):** emit one note per *memory* too (so mentions become real wikilinks and the
  graph includes memory↔entity edges), or keep memories as plain references for v1?
- [ ] **OQ-4 (enrichment source):** read the slim `GraphView` (simple, already cap-free) or the richer
  `/api/ontology/*` (carries reason/strength/confidence/mentions)? The note body wants the rich fields.
- [ ] **OQ-5 (reconciliation):** delete stale notes vs. move them to `_archive/` — which is least
  surprising when a user has clicked into a note that later vanishes from the graph?
