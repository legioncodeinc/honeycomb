# `get.theapiary.sh` — the Honeycomb install surface

A self-contained **Cloudflare Pages** project that serves the one-command Honeycomb
installer at **https://get.theapiary.sh** with content negotiation, plus an
"inspect before you pipe" page and published SHA-256 checksums.

This resolves the PRD-050a non-gating follow-up (vanity domain + published checksum +
inspect page). It **consumes** PRD-048 — the page installs the published npm package
`@legioncodeinc/honeycomb`; it does not publish it.

---

## What's here

```
site/install/
├── build.mjs              # DRY build: copies canonical scripts, hashes them, renders the page
├── index.template.html    # the "inspect before piping" page template (checksums injected at build)
├── functions/
│   └── index.js           # Pages Function: GET / content-negotiates shell-vs-browser
├── _headers               # pins text/plain + nosniff on /install.sh /install.ps1 /SHA256SUMS
├── README.md              # this file
└── dist/                  # GENERATED by build.mjs — gitignored, never committed
    ├── install.sh         # copy of scripts/install/install.sh (single source of truth)
    ├── install.ps1        # copy of scripts/install/install.ps1
    ├── SHA256SUMS         # sha256sum -c — verifiable checksums of the two scripts above
    └── index.html         # rendered inspect page with live checksums
```

**Single source of truth:** the installer scripts live ONCE at
[`scripts/install/`](../../scripts/install/). `build.mjs` copies them into `dist/` and
computes the checksums from exactly those copies, so what's served is always byte-identical
to what's checksummed. Never edit the copies in `dist/`; edit `scripts/install/` and rebuild.

## Content negotiation (the `get.pnpm.io` / `sh.rustup.rs` pattern)

| Request | Served | Why |
|---|---|---|
| `GET /` from a **shell** (curl/wget/PowerShell — `Accept` lacks `text/html`, or a CLI User-Agent) | `install.sh` as `text/plain` | so `curl -fsSL https://get.theapiary.sh \| sh` works |
| `GET /` from a **browser** (`Accept: text/html`) | the inspect `index.html` | a human gets the readable page |
| `GET /install.sh`, `/install.ps1`, `/SHA256SUMS` | always `text/plain` (`_headers`) | read/verify before running; `irm …/install.ps1 \| iex` works |

The negotiation lives in `functions/index.js` (`onRequest`). It only governs `/`; the explicit
file paths are plain static assets governed by `_headers`. The Function re-wraps the root shell
response to **guarantee** `text/plain; charset=utf-8` regardless of asset defaults.

## Cloudflare dashboard setup (operator steps)

### 1. Create the Pages project

Cloudflare dashboard → **Workers & Pages** → **Create** → **Pages** → **Connect to Git** →
select the `legioncodeinc/honeycomb` repo. Then set:

| Setting | Value |
|---|---|
| **Production branch** | a **release tag/branch**, not `main` (see "Version pinning" below) |
| **Root directory** (Build settings → "Path") | `site/install` |
| **Framework preset** | `None` |
| **Build command** | `node build.mjs` |
| **Build output directory** | `dist` |
| **Functions directory** | `functions` (auto-detected when present under the root dir) |

> The root directory is `site/install`, so the build command runs *inside* `site/install`
> and `build.mjs`'s relative paths (`../../scripts/install/…`) resolve to the repo root.
> No environment variables are required. Node 18+ is the Cloudflare build default — `build.mjs`
> uses only stable `node:` builtins, so any current build image works.

### 2. Add the custom domain `get.theapiary.sh`

Pages project → **Custom domains** → **Set up a custom domain** → enter `get.theapiary.sh`.

- **If the `theapiary.sh` zone is already on Cloudflare** (nameservers point at Cloudflare):
  the custom-domain flow **auto-provisions** the DNS record and the TLS cert. Click through and
  wait for "Active". Nothing else to do.
- **If `theapiary.sh` is NOT on Cloudflare:** add this DNS record at your DNS provider, then
  return to the dashboard and confirm:

  ```
  Type:   CNAME
  Name:   get            (i.e. get.theapiary.sh)
  Target: <your-project>.pages.dev
  Proxy:  (CNAME flattening / as your provider supports)
  ```

  Cloudflare will issue the cert once the CNAME resolves. Apex/`theapiary.sh` is unaffected.

### 3. Verify after deploy

```sh
# shell client gets the script as plain text
curl -fsSL https://get.theapiary.sh | head -3
curl -fsSLI https://get.theapiary.sh | grep -i content-type      # text/plain; charset=utf-8

# explicit paths are plain text + nosniff
curl -fsSLI https://get.theapiary.sh/install.ps1 | grep -i 'content-type\|nosniff'

# checksums verify against the served scripts
curl -fsSL -O https://get.theapiary.sh/install.sh
curl -fsSL -O https://get.theapiary.sh/SHA256SUMS
sha256sum -c SHA256SUMS                                            # install.sh: OK

# a browser Accept header gets the HTML page instead
curl -fsSL -H 'Accept: text/html' https://get.theapiary.sh | grep -i '<title>'
```

## Version pinning (deploy from a tag, not a moving branch)

**Deploy the Pages project from a git tag / release, not from `main`.** Reasons:

1. **Immutable served script.** Pointing the production deployment at a release tag means the
   script served at `https://get.theapiary.sh` is frozen for that release — it can't silently
   change under users between deploys.
2. **Stable checksum.** Because `build.mjs` hashes the exact scripts at build time, a pinned tag
   produces a stable `SHA256SUMS`. Anyone who recorded a checksum can re-verify it later.
   If production tracked `main`, every merge would rotate the published checksum and break
   "verify the hash I saved yesterday".

To cut a new install-surface release: tag the repo (the same tag you publish the npm package
from), then point the Pages project's production deployment at that tag. The checksum on the page
updates to match the newly served scripts.

## Why `site/` never ships in the npm tarball

The npm package uses a `files` **allowlist** in `package.json` — only the paths it names are
packed. `site/` is **not** in that allowlist, so the Pages project, this README, the template,
the Function, and `dist/` are all excluded from `npm pack` / publish by construction (verified
with `npm pack --dry-run`). No `.npmignore` entry is needed; the allowlist is the gate.
